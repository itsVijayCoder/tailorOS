WhatsApp Chat Connector

WhatsApp Chat Connector is a separate multi-tenant messaging product that TailorOS uses for order confirmations, receipts, trial reminders, ready-for-pickup alerts, balance reminders, delivery status, and customer chat. It starts with Meta WhatsApp Cloud API direct for cost control and is designed with provider adapters so Twilio, 360dialog, or Indian BSPs can be added later.

Prepared July 3, 2026
  • Recommended v1 provider: Meta WhatsApp Cloud API direct.
  • Recommended architecture: reusable connector, not TailorOS-only code.
  • Primary messages: utility and service messages, not marketing campaigns.
  • Multi-client design: each tenant uses its own credentials, templates, phone number, and usage ledger.

Product decision

Build WhatsApp as a reusable connector product

Production-grade recommendation

Do not build WhatsApp as a small feature inside TailorOS. Build WhatsApp Chat Connector as a separate tenant-aware product with its own channel accounts, credential vault, template registry, message orchestrator, webhook receiver, conversation router, usage ledger, and support console. TailorOS should emit business events; the connector should handle WhatsApp policy, provider routing, delivery status, inbound chat, and cost tracking.

Why separate

A separate connector can serve TailorOS first and later support SalonOS, RepairOS, ClinicOS, or any other SMB product without rebuilding WhatsApp each time.

Why Meta direct first

Meta direct gives the clearest long-term cost control and official Cloud API behavior. It requires more setup discipline, but it avoids early provider lock-in.

When to add BSPs

Add Twilio, 360dialog, WATI, Interakt, Authkey, or another BSP only when onboarding, managed inbox, local support, or client preference justifies the extra layer.

Scope boundary

What the connector includes and excludes

In scope for v1

  • Multi-tenant setup for many client shops and future product installations.
  • Meta Cloud API direct account connection and phone number mapping.
  • Credential vault with secret references, token status, rotation metadata, and support audit.
  • Template registry for order confirmation, receipt, trial reminder, ready for pickup, payment due, delivery, opt-in, and opt-out flows.
  • Outbound utility template sending from TailorOS events.
  • Inbound webhook receiver for customer messages, delivery statuses, failures, opt-outs, and provider events.
  • Conversation routing for STATUS, RECEIPT, DUE, HELP, shared-mobile disambiguation, and staff handoff.
  • Usage ledger by tenant, product, country, message category, provider, template, and month.

Out of scope for v1

  • Bulk marketing campaigns and festival promotions.
  • AI agent automation as the default support experience.
  • Voice calling, WhatsApp Business Calling, and IVR.
  • Full CRM inbox replacement for every sales/support use case.
  • Embedded Signup until manual onboarding becomes a proven bottleneck.
  • Provider marketplace with many adapters on day one.
  • Customer portal login; WhatsApp chat remains the customer-facing channel.
  • Exact long-term pricing guarantees because provider and Meta rates can change.

Provider strategy

Cost-efficient and scalable WhatsApp options

Provider option Best use case Cost and ease Strengths Recommendation
Meta Cloud API direct Default SaaS path where you can manage setup, tokens, templates, webhooks, and support. Most cost-efficient long term because there is no additional provider markup beyond Meta charges, but setup is more operationally demanding. Official API, direct control, best for multi-client platform architecture, strongest long-term ownership. Use first. Build v1 on Meta direct with clean provider abstraction.
Twilio WhatsApp Teams that want a developer-friendly CPaaS, possible SMS fallback, or existing Twilio stack. Easy to use. Twilio states WhatsApp pricing includes Twilio's per-message fee plus Meta template fees passed through to customers. Strong docs, sandbox, logs, developer tooling, SMS/voice ecosystem. Add as optional adapter after Meta direct if clients need Twilio ecosystem.
360dialog Clients or agencies wanting WhatsApp-focused BSP onboarding and API access. Can simplify WABA onboarding, but pricing and support terms must be rechecked during vendor selection. WhatsApp-focused provider model, potential reseller fit. Evaluate for onboarding scale after pilot shops.
WATI, Interakt, Authkey, Indian BSPs Non-technical SMB clients who want managed inbox, local support, campaigns, or ready-made support tools. Easier for some clients but can add subscription fees, provider lock-in, and API differences. Local support, campaign UI, team inbox, support tooling, faster managed onboarding. Use only when a client needs managed features more than platform control.

Final provider choice

For TailorOS v1, use Meta Cloud API direct. Build the connector so provider adapters are isolated. This gives the lowest long-term cost path while preserving the option to add Twilio, 360dialog, or an Indian BSP later for specific client segments.

Architecture

Connector responsibilities

Core components

  • Channel account manager: maps tenant, product installation, provider, WABA, phone number ID, display number, quality rating, and messaging limit tier.
  • Credential vault: stores secret references, token status, app secret references, verify token references, expiry, and rotation metadata.
  • Template registry: maps internal template purposes to provider template names, languages, categories, variables, approval status, and fallback instructions.
  • Message orchestrator: checks opt-in, opt-out, quiet hours, template status, category, rate limits, and idempotency before sending.
  • Provider adapter: translates internal send requests and webhook events to Meta, Twilio, 360dialog, or BSP-specific behavior.

Operational components

  • Webhook receiver: validates provider webhooks, normalizes events, deduplicates retries, and routes to the correct tenant.
  • Conversation router: routes inbound messages to command bot, TailorOS lookup, or staff handoff.
  • Usage ledger: records delivered messages by tenant, product, country, category, provider, and month.
  • Support console: helps platform admins debug failed sends, connection status, template status, and account health.
  • Policy guard: blocks disallowed marketing, missing consent, wrong template category, sensitive media, and opted-out recipients.

Core flows

End-to-end messaging workflows

Outbound order confirmation

  1. TailorOS emits ORDER_BOOKED with tenant, product, order, customer, contact method, language, and idempotency key.
  2. Connector finds the tenant's channel account and approved order confirmation template.
  3. Connector checks opt-in, opt-out, quiet hours, template status, and idempotency.
  4. Provider adapter sends through Meta Cloud API direct in v1.
  5. Webhook updates accepted, sent, delivered, read, or failed status in the message log.

Inbound customer status chat

  1. Customer sends STATUS, RECEIPT, DUE, HELP, or a free text question.
  2. Webhook receiver validates and normalizes the inbound message.
  3. Conversation router maps phone number to tenant and TailorOS family/customer context.
  4. If one active order exists, connector replies with status inside the service window.
  5. If multiple orders or family members exist, connector asks the customer to choose safely.

Template management

  1. Platform admin creates an internal template purpose such as ready for pickup.
  2. Tenant maps the purpose to approved provider templates in English, Tamil, or other supported languages.
  3. Template status is tracked as draft, pending, approved, rejected, paused, or disabled.
  4. Rejected templates require fallback actions and revised copy.
  5. TailorOS can only trigger approved template purposes for that tenant.

Opt-out and policy safety

  1. Customer sends STOP or a shop-approved local opt-out phrase.
  2. Connector records opt-out on the contact method and conversation.
  3. Future non-required messages are blocked automatically.
  4. Staff sees opt-out state inside TailorOS before attempting a send.
  5. Owner can view opt-out rates and failed sends by template.

Internal API contracts

How TailorOS and future products call the connector

These are product contracts, not code. They define the interface TailorOS expects from the connector.

Internal API Purpose Required information Expected behavior
POST /messages/template Send approved template message. tenant_id, product_id, recipient, template_purpose, language, variables, idempotency_key. Validate consent, template status, category, and idempotency before provider send.
POST /messages/service Reply inside customer service window. tenant_id, conversation_id, recipient, message body, source product, staff user if manual. Allow only inside open service window; otherwise recommend template path.
GET /conversations/{id} Show conversation in TailorOS. tenant_id, conversation_id, customer reference, order reference if known. Return normalized messages, status, assigned staff, and open window timing.
POST /webhooks/{provider} Receive provider events. Provider signature, account identifiers, message ID, event type, raw payload reference. Validate, dedupe, normalize, route, and store event history.
GET /usage Show usage and cost evidence. tenant_id, product_id, month, provider, category, country. Return delivered count, failed count, estimated cost, billed state, and top templates.
POST /contacts/consent Record opt-in or opt-out. tenant_id, contact method, customer reference, consent state, source, timestamp. Update message eligibility and write audit trail.

Pricing and cost controls

How cost should be handled

Current platform assumptions

  • Meta states WhatsApp Business Platform pricing is based on delivered messages, market, and message category.
  • Meta categories include marketing, utility, authentication, and service.
  • Service messages happen in a 24-hour customer service window after the user messages the business.
  • Utility messages sent in response to users can be favorable compared with business-initiated marketing.
  • Rates must be rechecked before launch because pricing pages and rate cards can change.

Connector cost controls

  • Default TailorOS sends: order confirmation, receipt, trial reminder, ready for pickup, payment due.
  • Do not send every production status change to the customer.
  • Show monthly usage by tenant and template purpose.
  • Flag tenants with rising failures, opt-outs, or provider quality issues.
  • Keep marketing templates disabled until the product intentionally supports campaigns.
Scenario Estimated monthly message volume Cost risk Product response
Small tailor shop 100 orders, roughly 300 to 500 order-related messages. Support and onboarding cost may matter more than message cost. Bundle a small allowance and show usage only when close to limit.
Growing boutique 500 orders, roughly 1,500 to 2,500 messages. Unnecessary reminders and failed templates can create avoidable cost. Add usage dashboard, quiet hours, and per-template controls.
Multi-shop SaaS 5,000+ messages across many tenants. Cost evidence, tenant isolation, provider quality, and support load become important. Use usage ledger, billing export, provider health checks, and support console.

Milestone plan

Deep production milestone analysis

Each milestone should be independently testable with TailorOS-style events and real provider constraints, while avoiding early provider lock-in.

M1

Tenant, product installation, and channel foundation

Create the reusable product boundary so TailorOS is only the first connected product, not the only product.

Build scope

  • Tenant records and product installation records.
  • Channel account records for WhatsApp phone numbers.
  • Provider field with Meta direct as default.
  • Admin views for connected, pending, disabled, and failed account states.

Deep analysis

  • This milestone decides whether the connector can scale beyond TailorOS.
  • Product installation is essential because one tenant may later use multiple products.
  • Channel accounts must not be shared across unrelated shops.

Done when

  • Platform admin can create tenant and product installation.
  • A tenant can have one or more WhatsApp channel accounts.
  • Each channel shows provider, WABA, phone number, status, and owner.
  • TailorOS can be registered as a source product.
M2

Credential vault and Meta direct connection

Safely connect Meta Cloud API credentials without exposing tokens to frontend users or other tenants.

Build scope

  • Credential metadata with secret references, not raw token display.
  • Meta business ID, WABA ID, phone number ID, app ID, and token status fields.
  • Connection health state and last check timestamp.
  • Audit log for credential creation, rotation, disable, and reconnect.

Deep analysis

  • Credential mistakes are high-risk because they can leak client communication access.
  • Disconnect/reconnect should not delete historical conversations or usage records.
  • Support staff need enough visibility to debug without seeing secrets.

Done when

  • Admin can store and update Meta channel credential references.
  • Credentials are tenant-isolated.
  • Raw tokens never appear in logs or normal UI.
  • Connection status is visible to support admins.
M3

Template registry and TailorOS message catalog

Create controlled, approved template purposes before sending customer messages.

Build scope

  • Internal template purposes for order confirmation, receipt, trial reminder, pickup alert, payment due, delivery, opt-in, and opt-out.
  • Provider template mapping per tenant and language.
  • Status tracking: draft, pending, approved, rejected, paused, disabled.
  • Variable definition and preview screen.

Deep analysis

  • Template chaos creates policy failures and incorrect customer messages.
  • Purpose mapping keeps TailorOS independent from provider-specific template names.
  • Tamil and English templates should be reviewed by the shop owner before production use.

Done when

  • Admin can register approved template names for a tenant.
  • TailorOS can request a template purpose without knowing provider template name.
  • Unapproved templates are blocked with clear fallback reason.
  • Template variables are visible before go-live.
M4

Outbound message orchestration

Send approved utility templates from TailorOS events with policy and cost controls.

Build scope

  • POST /messages/template internal API.
  • Opt-in, opt-out, quiet hour, template status, and idempotency checks.
  • Message request, provider message, and status records.
  • Meta direct provider adapter for outbound templates.

Deep analysis

  • Accepted by API is not delivered. The connector must wait for webhook status.
  • Idempotency prevents duplicate customer messages from retries.
  • Message categories should be visible so utility and marketing do not mix.

Done when

  • TailorOS can request order confirmation and receipt send.
  • Connector rejects sends missing consent or approved template.
  • Provider message ID is stored.
  • Staff can see pending, sent, delivered, failed, or blocked state.
M5

Webhook receiver and delivery status normalization

Make provider events trustworthy, deduplicated, and visible to TailorOS and support staff.

Build scope

  • Provider webhook endpoint for Meta direct.
  • Signature validation and verify-token challenge support.
  • Routing by WABA, phone number ID, and tenant channel account.
  • Normalized event log for sent, delivered, read, failed, inbound, and opt-out.

Deep analysis

  • Webhooks can retry and arrive more than once; events must be idempotent.
  • Raw payload references are useful for support but must not expose sensitive data unnecessarily.
  • Tenant routing is the safety boundary that prevents cross-client leakage.

Done when

  • Webhook verification succeeds.
  • Duplicate webhook payloads do not create duplicate business events.
  • Message delivery status updates on the message log.
  • Invalid signatures are rejected and not processed.
M6

Inbound conversation router and command bot

Let customers ask for order status, receipt, and dues without requiring staff to answer every message manually.

Build scope

  • Conversation records with open service window timing.
  • Commands: STATUS, RECEIPT, DUE, HELP.
  • Shared-mobile disambiguation by active order or customer choice.
  • Staff handoff and assigned conversation state.

Deep analysis

  • Shared family numbers are the main customer data risk.
  • Bot replies should stay deterministic in v1; avoid AI-generated answers until data permissions are mature.
  • Service-window replies are cost-efficient and should be used for customer-initiated chat.

Done when

  • Customer can message STATUS and receive safe status for the right order.
  • Multiple matching orders trigger a choice prompt.
  • Unknown numbers get a helpful fallback.
  • Staff can take over a conversation from the support view.
M7

TailorOS integration and notification timeline

Connect business events from TailorOS to templates, conversations, and delivery status.

Build scope

  • Event intake for ORDER_BOOKED, RECEIPT_GENERATED, PAYMENT_RECEIVED, TRIAL_SCHEDULED, and ORDER_READY_FOR_PICKUP.
  • Template purpose mapping for each event.
  • Callback or status API for TailorOS notification log.
  • Fallback reasons for blocked or failed sends.

Deep analysis

  • TailorOS should not need provider knowledge. It should ask for a business message purpose.
  • Failures must be visible where staff work: order and customer pages.
  • Event payload should include idempotency key to avoid duplicate sends.

Done when

  • TailorOS order booking can trigger an approved WhatsApp confirmation.
  • Delivery status returns to TailorOS notification log.
  • Blocked sends show clear reasons to staff.
  • Customer opt-out state updates TailorOS contact method.
M8

Usage ledger, cost reporting, and client billing evidence

Give owners and the platform enough visibility to control WhatsApp cost and bill fairly.

Build scope

  • Usage records by tenant, product, provider, country, category, template, and month.
  • Owner usage view with delivered, failed, and blocked messages.
  • Platform admin cost export and billing state.
  • Alerts for high failure rate, opt-out spike, or unusual cost growth.

Deep analysis

  • Usage evidence is required before charging clients for a message wallet or allowance.
  • Cost should be estimated but never treated as a guaranteed provider bill until reconciled.
  • Category-level reporting prevents marketing costs from hiding inside utility workflows.

Done when

  • Owner can see monthly WhatsApp usage.
  • Platform admin can export tenant usage.
  • Usage records distinguish delivered, failed, and blocked sends.
  • Top templates and high-cost categories are visible.
M9

Admin support console and production controls

Make onboarding and support reliable before expanding beyond the first few shops.

Build scope

  • Connection health dashboard.
  • Template approval and rejection tracking.
  • Failed message troubleshooting view.
  • Consent and opt-out audit.
  • Credential change audit and support access reasons.

Deep analysis

  • Support visibility reduces manual debugging and client frustration.
  • Credential and template changes must be auditable because they affect customer communication.
  • Admins should see enough to debug without exposing raw secrets or unrelated tenant data.

Done when

  • Support can identify why a message failed.
  • Support can see whether template, consent, provider, or webhook caused the issue.
  • Credential changes are audited.
  • Tenant data remains isolated during support.
M10

Embedded Signup and provider adapters planning

Prepare scale paths without delaying the first production release.

Build scope

  • Manual onboarding checklist for first shops.
  • Readiness assessment for Meta Embedded Signup.
  • Provider adapter interface documented for Twilio, 360dialog, and BSPs.
  • Decision matrix for when a tenant should use Meta direct versus provider adapter.

Deep analysis

  • Embedded Signup is useful only after manual onboarding becomes the bottleneck.
  • Adding too many providers early creates support complexity.
  • The adapter boundary should be tested with one future provider, not many partial integrations.

Done when

  • Platform has a clear manual onboarding process.
  • Future provider adapter requirements are documented.
  • Meta direct remains the production provider for v1.
  • Twilio or BSP adapter can be prioritized with a business reason.
M11

Production hardening, policy, and launch readiness

Stabilize the connector for real client use, support, compliance, and scale.

Build scope

  • Policy guardrails for marketing, utility, service, opt-out, sensitive media, and customer data leakage.
  • Monitoring for webhook failures, provider errors, high retry volume, and usage spikes.
  • Retention policy for raw payloads, media, message logs, and audit logs.
  • Go-live checklist for each tenant.

Deep analysis

  • Messaging products fail in production through edge cases: duplicate events, wrong tenant routing, missing consent, and unclear failures.
  • Retention should balance debugging needs with privacy.
  • Marketing should stay disabled until the connector has strong consent and category controls.

Done when

  • A tenant can go live using a checklist.
  • Bad webhook signatures are rejected.
  • Opted-out customers cannot receive non-required messages.
  • Support can monitor delivery health and usage.

High-risk mistakes

  • Putting provider tokens in frontend code or visible logs.
  • Routing webhook events using phone number text only instead of provider account identifiers.
  • Assuming provider acceptance means delivered to customer.
  • Sending customer data to a shared family mobile without disambiguation.
  • Using marketing copy inside utility templates.
  • Building provider-specific logic directly inside TailorOS.

Launch controls

  • Backend-only credential use with audit trail.
  • Tenant isolation by WABA and phone number ID.
  • Idempotency keys for every outbound business event.
  • Webhook signature validation and duplicate handling.
  • Manual fallback paths for failed or blocked sends.
  • Recheck Meta and provider pricing before client billing launch.
