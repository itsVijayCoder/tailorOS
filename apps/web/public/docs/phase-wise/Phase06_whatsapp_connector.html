Phase 06 · WhatsApp Connector

WhatsApp Connector implementation objective

Phase 06 implements the separate multi-tenant WhatsApp Connector. TailorOS emits business events; the connector validates policy, maps templates, sends via Meta Cloud API direct, receives webhooks, deduplicates events, updates delivery status, and routes inbound customer chat.

Boundary rule. TailorOS should not know Meta template names, access tokens, phone number IDs, webhook signatures, provider pricing rules, or provider retry behavior. Those belong to the connector.

  • Internal send APIs require service binding or signed internal service token.

Phase 06 · WhatsApp Connector

Provider adapter and Meta first implementation

Start with Meta Cloud API direct for cost control and official behavior. Keep Twilio, 360dialog, Authkey, WATI/Interakt-style BSPs behind an adapter interface.

export interface WhatsAppProviderAdapter {
  provider: 'meta' | 'twilio' | '360dialog' | 'authkey' | 'wati';
  sendTemplate(input: TemplateSendInput): Promise<ProviderSendResult>;
  sendTextWithinServiceWindow(input: TextSendInput): Promise<ProviderSendResult>;
  normalizeWebhook(input: ProviderWebhookInput): Promise<NormalizedWebhookEvent[]>;
  verifyWebhook?(request: Request, credential: ChannelCredential): Promise<boolean>;
}
Adapter ruleReason
No TailorOS imports in provider adapters.Connector must support future products.
Normalize provider errors into internal error codes.Support UI should not depend on provider-specific messages.
Store provider payload reference, not huge payload in hot tables forever.Controls D1 growth and improves search speed.

Phase 06 · WhatsApp Connector

Webhook, outbox, retries, and service window

WhatsApp webhooks can be duplicated and can arrive out of order. Queue consumers should be idempotent and use status ranking.

TailorOS order event → outbox_jobs insert → WHATSAPP_SEND_QUEUE → whatsapp-consumer sends through provider → wa_messages: accepted → Meta webhook status → webhook event dedupe → status rank update → TailorOS notification timeline callback/projection
  • Outbox job uses idempotency key.
  • Queue jobs retry transient provider failures.
  • DLQ stores terminal failures or repeated errors.
  • Webhook returns quickly and processes heavy work asynchronously.
  • Service-window replies are allowed only after customer message and within window; outside window use approved template.

Phase 06 · WhatsApp Connector

Shared mobile disambiguation and customer safety

A Tamil Nadu family may share one WhatsApp number. The connector must avoid exposing one family member’s order details to the wrong person without context.

Inbound caseResponse
One active order for the phone numberReply with safe order status and order code.
Multiple active orders across family profilesAsk customer to choose order code or profile name; show minimal list.
Unknown phone numberAsk them to contact shop or provide order ID; do not reveal data.
Customer asks for receiptSend receipt link only after order match; optionally ask mobile-last-4/order code.
STOP / opt-out phraseRecord opt-out and block non-required future messages.

Phase 06 · WhatsApp Connector

Tests and launch checklist

Tests

  • Template send with approved template succeeds and stores provider message ID.
  • Missing consent blocks send with reason.
  • Opt-out blocks non-required messages.
  • Duplicate webhook does not create duplicate message/event.
  • Out-of-order delivered/read statuses do not downgrade final message state.
  • Multiple active orders under same mobile trigger disambiguation.
  • Invalid webhook signature is rejected.

Exit checklist

  • Meta direct provider adapter implemented.
  • Credential encryption and rotation metadata in place.
  • Template mapping UI supports Tamil and English.
  • Queue + DLQ configured for send and webhook processing.
  • TailorOS notification timeline shows queued/sent/delivered/read/failed/blocked.
  • Usage ledger records message category, template, provider, status, and tenant.
