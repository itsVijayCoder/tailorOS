Phase 10 · Scale & Reliability

Scale, reliability, and future roadmap objective

Phase 10 prepares the product for growth beyond the first pilot: more tenants, larger tenant databases, cross-tenant analytics, realtime chat, offline drafts, cost controls, backup/restore, and migration paths to Postgres or specialized services when needed.

Scale principle. Do not scale by turning the MVP into a monolith. Scale by isolating tenants, projecting analytics, archiving cold logs, using queues for background work, and keeping escape hatches for large tenants.

Phase 10 · Scale & Reliability

Realtime and inbox scale path

Start with polling for the staff inbox and order board. Add Durable Objects when live collaboration matters.

StageApproachUse case
MVPTanStack Query polling every few seconds.Simple inbox and order status updates.
GrowthDurable Object per tenant inbox room.Live chat, typing, assignment, presence.
AdvancedDurable Object per order board / branch.Realtime production board and multi-tab sync.
  • Do not add WebSockets before there is real chat volume.
  • Keep persistent message history in D1, not in Durable Object memory only.
  • Durable Object manages live state; D1 remains source of truth.

Offline and poor-network support

Many small shops may have unreliable internet. Do not promise full offline-first in v1, but design for offline drafts.

Phase A: local autosave drafts for order/measurement forms Phase B: visible offline indicator + retry queue for drafts Phase C: background sync with conflict detection Phase D: limited offline order creation with server reconciliation
ConflictResolution
Measurement changed on another device while draft open.Show version conflict and ask whether to use latest, keep draft as override, or create new version.
Payment already recorded on another device.Server rejects duplicate/invalid balance; staff reviews current balance.
Order status changed while offline.Apply only valid state transitions; otherwise show conflict.

Phase 10 · Scale & Reliability

Cost and usage controls

  • Store R2 object count and storage estimate per tenant.
  • Track D1 rows read/written per critical route to identify expensive screens.
  • WhatsApp usage ledger by tenant/month/template/category/status.
  • Queue DLQ metrics and retry counts by job type.
  • Archive raw webhook payloads after retention period.
  • Use plan limits: users, monthly orders, media storage, WhatsApp messages, exports.

Escape hatches and future architecture

A senior design keeps migration paths open. D1 is excellent for Cloudflare-first operational SaaS, but some tenants or features may eventually need Postgres, search service, or analytics warehouse.

Future needMigration pathPreparation now
Large relational reportingPostgres via Hyperdrive or separate GCP/Neon/Supabase service.Keep repositories behind interfaces.
Advanced fuzzy/Tamil transliteration searchTypesense/Meilisearch/Postgres trigram later.Use search adapter interface and projection events.
Heavy PDF rendering/OCR/image processingSmall Cloud Run or worker-compatible external service.Queue jobs and R2 file contracts.
Multi-product WhatsApp platformExtract connector into separate repo/service.Keep product installation boundary.

Phase 10 · Scale & Reliability

Scale readiness checklist

Scale tests

  • 100 tenant provisioning dry run in staging or simulated environment.
  • Tenant migration fan-out with retry and DLQ.
  • Search benchmark with 50k orders in one tenant.
  • WhatsApp webhook burst test with duplicate and out-of-order events.
  • R2 signed URL stress for receipt downloads.
  • Backup/restore drill using D1 Time Travel and tenant export.

Exit checklist

  • Tenant storage and usage dashboard exists.
  • Projection pipeline exists for platform-wide analytics/search.
  • Runbooks tested, not just written.
  • Large tenant migration plan documented.
  • Cost thresholds and alerts configured.
  • Roadmap separates v1.5 enhancements from v1 stability fixes.
