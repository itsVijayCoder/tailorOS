Phase 07 · Search & Speed

Search and speed objective

Phase 07 makes the app feel fast. The two most important speed features are command search and instant screen feedback. Staff should find a customer, order, receipt, or WhatsApp conversation without knowing which module to open first.

Performance rule. Optimize the actual shop path: mobile search → profile selection → order/measurement → payment/receipt. Do not over-optimize decorative dashboards while the counter flow is slow.

Phase 07 · Search & Speed

Tenant-local global search model

Because each tenant has a separate D1 database, “global search” inside TailorOS means tenant-wide search. Platform-wide admin search should use a projection DB, not live fan-out queries to every tenant DB.

CREATE TABLE search_projection (
  id TEXT PRIMARY KEY,
  entity_type TEXT NOT NULL,
  entity_id TEXT NOT NULL,
  title TEXT NOT NULL,
  subtitle TEXT,
  mobile_e164 TEXT,
  customer_code TEXT,
  order_code TEXT,
  receipt_code TEXT,
  status TEXT,
  delivery_date TEXT,
  updated_at TEXT NOT NULL
);

CREATE VIRTUAL TABLE search_docs USING fts5(
  entity_type,
  entity_id UNINDEXED,
  title,
  subtitle,
  body,
  tags,
  tokenize = 'unicode61'

Phase 07 · Search & Speed

D1 query and index discipline

Do not rely on FTS for everything. Use normal indexes for hot exact lookups and FTS for name/item/free text. Periodically review D1 query plans and rows read.

  • Use EXPLAIN QUERY PLAN during development for every list/search query.
  • Use multi-column indexes in the same order as query predicates.
  • Avoid offset pagination on large lists; use cursor/keyset pagination.
  • Keep WhatsApp/audit logs out of normal dashboard queries.
  • Use D1 Sessions when read replication is enabled and the user expects read-after-write consistency.
CREATE INDEX idx_orders_status_delivery
  ON orders(current_status, promised_delivery_date);

CREATE INDEX idx_orders_customer_date
  ON orders(customer_profile_id, order_date DESC);

CREATE INDEX idx_search_mobile
  ON search_projection(mobile_e164);

Phase 07 · Search & Speed

Platform-wide search and analytics projection

Per-tenant D1 is excellent for isolation, but it prevents easy SQL joins across all tenants. For support/admin search, use a projection pipeline.

Tenant write → search_projection_update queue event → platform_projection_worker → CONTROL_SEARCH_DB / analytics D1 → support admin search Projection content: tenant_id, entity_type, entity_id, display_title, status, last_activity, non-sensitive summary
  • Do not copy sensitive measurement values into global search projection.
  • Store enough to help support find the tenant/order, then require scoped support access.
  • Projection updates must be idempotent and replayable.

Phase 07 · Search & Speed

Performance tests and checklist

Performance budgets

  • Dashboard initial useful render: under 2 seconds on normal 4G after login.
  • Command palette opens in under 120ms on cached UI.
  • Mobile/order exact search backend under 100ms for pilot-scale tenant.
  • Customer detail page avoids loading full order/payment/message history by default.
  • Public receipt page loads with minimal JS.

Tests

  • Seed tenant DB with 10k contacts, 30k profiles, 50k orders; benchmark search.
  • Verify query plans use indexes.
  • Simulate 10 quick search inputs and ensure stale responses do not overwrite newer results.
  • Test read-after-write after enabling read replication with D1 Sessions.
  • Run Lighthouse/WebPageTest for dashboard and receipt pages.

Exit checklist

  • Command search implemented.
  • FTS projection updated on customer/order/receipt/message changes.
  • Performance budgets documented in CI or QA checklist.
  • Large tables use pagination/virtualization.
  • Motion uses transform/opacity only.
