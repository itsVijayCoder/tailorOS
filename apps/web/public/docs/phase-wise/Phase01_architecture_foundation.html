Phase 01 · Foundation

Phase objective and architectural decision

Phase 01 creates the engineering foundation for a product that can support TailorOS and the WhatsApp Connector as separate but connected systems. The goal is not to build screens first; the goal is to establish boundaries, naming, packages, runtime patterns, validation, and deployment discipline so every later phase remains maintainable.

Primary decision. Use a TypeScript monorepo with a Next.js web app, Hono API Workers, a control-plane Worker, a tenant API Worker template, WhatsApp Connector Workers, shared domain packages, and Cloudflare bindings. This follows SOLID by separating product, domain, infrastructure, and integration concerns.
1source of truth for schemas
4core deployables before product modules

Design principles

  • Single Responsibility: web routes render UI; API Workers validate and orchestrate; repositories only access D1; domain services enforce business rules.
  • Open/Closed: WhatsApp providers, storage adapters, and search adapters should be extensible through interfaces without changing TailorOS order code.
  • Liskov and Interface Segregation: small provider interfaces such as sendTemplate(), normalizeWebhook(), and downloadMedia() prevent giant all-purpose adapter contracts.
  • Dependency Inversion: domain services receive ports such as OrderRepository, MessageOutbox, and FileStore; they do not import Cloudflare runtime directly.
  • DRY with boundaries: share schemas and domain rules, not entire feature implementations. Duplicating UI layout occasionally is better than forcing a generic component that hides business meaning.

Target architecture and component boundaries

Browser / PWA → Next.js Web on Cloudflare Workers → API Gateway Worker → Control Plane Worker → Tenant Dispatch Worker → Tenant API Worker with TENANT_DB binding → WhatsApp Connector Worker → WhatsApp Queue Consumer → Meta WhatsApp Cloud API Data services: CONTROL_DB: tenants, users, memberships, provisioning, tenant DB registry TENANT_DB: contacts, profiles, measurements, orders, payments, receipts, messages R2: receipts, measurement photos, garment reference images, tenant exports Queues: provisioning, migrations, WhatsApp send, webhook processing, receipt jobs

The API Gateway must not contain tailoring business logic. It should resolve tenant, authenticate user, rate-limit, validate route shape, and dispatch to the correct service. TailorOS business logic belongs in tenant services. WhatsApp delivery logic belongs in the connector. Provisioning logic belongs in the control plane.

Repository structure and coding standards

Use a monorepo because the product is young and benefits from shared schemas, shared domain rules, and a single CI gate. Keep deployable applications under apps/ and reusable logic under packages/.

tailoros/
  apps/
    web/                     # Next.js staff app, receipt public pages, onboarding UI
    api-gateway/             # public API entry and tenant dispatch
    control-plane/           # tenant provisioning, migrations, platform admin
    tenant-api-template/     # TailorOS tenant API code deployed/bound per tenant
    whatsapp-connector/      # Meta webhooks, template send API, provider adapters
    whatsapp-consumer/       # queue consumers for outbound and inbound work
    realtime/                # later Durable Objects for inbox/order board
  packages/
    schemas/                 # Zod contracts shared by UI, API, queues, tests
    core/                    # domain rules, IDs, phone, money, dates, state machines
    db-control/              # control DB migrations + repositories
    db-tenant/               # tenant DB migrations + repositories
    ui/                      # shadcn-based design system components
    search/                  # search normalization, FTS builders, projection events
    whatsapp/                # provider interfaces, message builders, policy
    r2/                      # object keys, signed URL policies, upload contracts
    observability/           # logger, request IDs, metrics helpers
    test-utils/              # factories, fixtures, Worker test helpers
  infra/cloudflare/
    wrangler/                # wrangler.jsonc per deployable
    scripts/                 # create DB, provision tenant, migrate all, export/delete
  docs/
    architecture/            # ADRs
    runbooks/                # support and incident procedures

Required engineering rules

  • Every API input must have a Zod schema in packages/schemas.
  • Every database write must pass through a repository or a transaction helper, never ad hoc SQL in route handlers.
  • Every domain operation must emit audit metadata: actor, tenant, action, entity, request ID, timestamp.
  • Every queue payload must be versioned: { type, version, idempotencyKey, payload }.
  • Routes may call services; services may call repositories; repositories may call D1. Do not invert this flow.

Phase 01 · Foundation

Deliverables, tests, and acceptance checklist

Deliverables

  • Monorepo initialized with TypeScript, pnpm, Turbo, ESLint, Prettier, path aliases, and strict TS config.
  • Five minimal deployables compile: web, api-gateway, control-plane, tenant-api-template, whatsapp-connector.
  • Shared Zod schemas and API response format are available.
  • Cloudflare local dev with Wrangler and Miniflare works for at least one test tenant.
  • ADRs created for Cloudflare stack, per-tenant D1, WhatsApp Connector, and design system.

Automated tests

  • Unit: phone normalization, ID generation, money calculations, order state machine.
  • Worker: Hono route health, request ID middleware, error handler, service binding mock.
  • Contract: API success/error format and queue payload versions.
  • Architecture: dependency boundaries checked with lint rules or dependency-cruiser.

Exit checklist

  • No deployable imports from another deployable directly; shared code lives in packages.
  • No tenant-specific data is stored in global config files.
  • No secrets are committed or printed in logs.
  • Every route returns a request ID.
  • Every business package is testable without Cloudflare runtime.
