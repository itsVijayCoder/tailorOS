Phase 05 · Core Product

TailorOS core module implementation objective

Phase 05 builds the actual shop operating system screens and APIs: dashboard, family-aware customer directory, measurements, order book, production board, payments, receipts, reports, and settings. This phase should produce a usable pilot product for one real tailor shop.

Product rule. Build the shop workflow before advanced automation. WhatsApp should amplify clean TailorOS data; it should not compensate for weak order, measurement, and payment records.

Phase 05 · Core Product

Customer and family module

Customer lookup is the adoption-critical flow. It must be faster than opening a notebook.

  1. Staff taps search or presses Ctrl + K.
  2. Input can be mobile, name, customer code, order code, receipt code.
  3. If mobile has multiple family profiles, show all profiles as selectable cards.
  4. Staff must select exact profile before viewing/editing measurements.
  5. Customer timeline shows orders, measurements, payments, receipts, and WhatsApp messages.

Measurement center

Phase 05 · Core Product

Order book and new order wizard

The new order wizard should be optimized for counter speed and correctness.

Step 1: Search/select contact and exact customer profile Step 2: Add one or more garment items Step 3: Select measurement version or capture new measurement Step 4: Add item-level override if needed Step 5: Set trial date, delivery date, price, discount Step 6: Record advance payment Step 7: Generate receipt and optional WhatsApp confirmation
  • Repeat customer flow target: around 2 minutes.
  • New customer flow target: under 5 minutes after staff training.
  • Wizard saves local draft to prevent loss on accidental navigation.
  • Order can be saved as draft only if required fields are incomplete.
  • Order ID and item IDs are generated server-side.

Phase 05 · Core Product

Payments, receipts, and reports

Payments and receipts are trust-sensitive. Do not allow silent mutation. Use ledger records, correction reasons, and audit logs.

  • Payment modes: cash, UPI, card, bank, credit, adjustment, refund.
  • Receipt generated as HTML-first printable page; PDF generation can be added later.
  • R2 stores receipt snapshots and downloadable files.
  • Receipt public link uses signed token and expiry policy.
  • Reports include today collection, pending balance, overdue delivery, tailor workload, and WhatsApp delivery failures.

Phase 05 · Core Product

Tests and acceptance checklist

E2E pilot scenarios

  • Create new contact with 4 family members under one mobile number.
  • Create blouse measurement for one profile and shirt measurement for another profile.
  • Create order with two items, one ready and one still stitching.
  • Record advance and balance payments; correct a wrong payment with reason.
  • Generate receipt and verify customer name, order ID, item summary, paid amount, balance.
  • Search mobile, name, customer code, order code, and receipt code.

Exit checklist

  • Shop can run daily order entry without notebook for active orders.
  • Owner can open dashboard and know today's work in under 30 seconds.
  • Every critical edit is audited.
  • Every screen has empty/error/loading states.
  • Mobile layout is usable at counter with one hand.
