Phase 03 · Control Plane

Control plane and per-tenant D1 objective

Phase 03 builds the tenant lifecycle: registration, tenant registry, per-tenant D1 database creation, migration, seeding, activation, suspension, export, delete, and support tooling. This is the foundation for a SaaS where every tailor shop receives its own operational database.

Important architecture rule. D1 per-tenant databases are a scale-out pattern, but the app still needs a control plane. The control DB tracks which shop owns which D1 database, which worker handles requests, which schema version is active, and whether provisioning or migration is in progress.

Phase 03 · Control Plane

Control-plane schema

The control database should contain platform-level records only. Do not store customer measurements, order items, payments, or chat messages in the control DB.

CREATE TABLE tenants (
  id TEXT PRIMARY KEY,
  tenant_code TEXT NOT NULL UNIQUE,
  slug TEXT NOT NULL UNIQUE,
  business_name TEXT NOT NULL,
  city TEXT,
  state TEXT DEFAULT 'Tamil Nadu',
  timezone TEXT NOT NULL DEFAULT 'Asia/Kolkata',

  plan_code TEXT NOT NULL DEFAULT 'pilot',
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL

Phase 03 · Control Plane

New tenant provisioning state machine

requested → validating_owner → tenant_reserved → db_creating → db_migrating → defaults_seeding → worker_registering → health_checking → active recoverable failures: → failed_db_create → failed_migration → failed_seed → failed_worker_register → needs_manual_review

Provisioning must be asynchronous. Do not create a D1 database, run migrations, seed templates, configure worker bindings, and send welcome notifications inside one HTTP request. Insert a job, return a provisioning screen, and let a queue consumer complete the steps.

export async function createTenantSignup(input: TenantSignupInput, env: Env) {
  const parsed = tenantSignupSchema.parse(input);
  await verifyTurnstile(parsed.turnstileToken, env);

  const tenantId = newId('ten');
  const slug = await reserveTenantSlug(env.CONTROL_DB, parsed.shopName, parsed.city);

  await env.CONTROL_DB.batch([
    env.CONTROL_DB.prepare(`INSERT INTO tenants (...) VALUES (...)`).bind(...),
    env.CONTROL_DB.prepare(`INSERT INTO provisioning_jobs (...) VALUES (...)`).bind(...)
  ]);

  await env.TENANT_PROVISION_QUEUE.send({
    type: 'TENANT_PROVISION_REQUESTED',
    version: 1,
    tenantId,
    idempotencyKey: `tenant-provision:${tenantId}`
  });

  return { tenantId, slug, status: 'provisioning_pending' };
}

Phase 03 · Control Plane

D1 creation and tenant worker binding strategy

For pilot shops, manual per-tenant Worker deployments can work. For scalable self-serve SaaS, use a dispatch namespace / Workers for Platforms style where the API Gateway resolves tenant and dispatches to the tenant Worker. Avoid one huge Worker with thousands of static D1 bindings because binding limits and deployment complexity become a bottleneck.

ScaleRecommended approachReason
1–5 pilot shopsManual tenant D1 + manual binding.Fastest to validate product, simplest debugging.
5–25 shopsAutomated D1 create + controlled worker deploy/register script.Reduces manual onboarding mistakes.
25+ shopsDispatch namespace / tenant-worker pattern.Cleaner isolation, operationally scalable.
Large analytics-heavy tenantDedicated tenant plan or Postgres migration path.D1 remains excellent for operational data but cross-tenant analytics need projections.

Phase 03 · Control Plane

Migration strategy and defaults

Every tenant database must track its schema version. Migrations should be small, forward-only by default, and repairable. Since tenants are isolated, a failed migration must affect only that tenant and must be visible in the control plane.

  • Apply migration files in numeric order and record schema_version.
  • Seed default garment templates separately from schema migration.
  • Use feature gates for tenant DBs not yet migrated.
  • Keep migration jobs idempotent so retrying does not duplicate defaults.
  • For risky changes, use expand → backfill → contract instead of destructive ALTERs.
CREATE TABLE tenant_schema_migrations (
  version INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  checksum TEXT NOT NULL,
  applied_at TEXT NOT NULL
);

CREATE TABLE tenant_defaults_seeded (
  seed_key TEXT PRIMARY KEY,
  seed_version INTEGER NOT NULL,
  applied_at TEXT NOT NULL
);

Phase 03 · Control Plane

Edge cases, tests, and acceptance checklist

Edge cases

CaseHandling
Duplicate shop slugReserve slug in a transaction-like control flow; suggest alternatives.
D1 create succeeds but migration failsMark tenant failed_migration, keep DB record, allow retry/rollback/manual review.
User refreshes signup pageUse idempotency key and return existing provisioning status.
Tenant activation before worker health checkDo not mark active until a health route validates schema/version/defaults.
Cloudflare API token permission issueFail provisioning with explicit reason and alert platform admin.
Tenant suspensionBlock API dispatch but preserve receipt links according to policy.

Tests

  • Signup creates tenant, provisioning job, and reserved slug.
  • Retrying same idempotency key does not create duplicate tenant or D1 database.
  • Migration failure records the failing step.
  • Inactive/suspended tenant cannot be dispatched.
  • Tenant DB health check confirms schema version and seed data.

Exit checklist

  • Control DB schema complete.
  • Queue-based provisioning flow implemented.
  • D1 per-tenant create path documented and tested in staging.
  • Tenant status dashboard available to platform admin.
  • Manual recovery runbook exists for every failure state.
