Phase 04 · Domain Model

Tenant domain model objective

Phase 04 builds the per-tenant operational database, repositories, domain services, and invariants for TailorOS. This phase converts the PRD into a durable data model: one mobile number can represent a family, each person has their own customer profile, measurements are versioned, orders have unique IDs, payments are ledger-based, and receipts/messages remain traceable.

Core invariant. Mobile number is a contact channel, not a customer identity. Never attach measurements directly to a phone number. Attach measurements to a customer profile and attach orders to a customer profile plus contact.

Phase 04 · Domain Model

Measurement versioning and order snapshots

Measurements are high-trust records. The system must preserve what the tailor used for a historical order even after a customer changes body size or style preference.

CREATE TABLE measurement_profiles (
  id TEXT PRIMARY KEY,
  customer_profile_id TEXT NOT NULL REFERENCES customer_profiles(id),
  garment_type_code TEXT NOT NULL,
  display_name TEXT NOT NULL,
  current_version_id TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE measurement_versions (
  id TEXT PRIMARY KEY,
  measurement_profile_id TEXT NOT NULL REFERENCES measurement_profiles(id),
  version_no INTEGER NOT NULL,
  values_json TEXT NOT NULL,
  unit TEXT NOT NULL DEFAULT 'inch',
  fit_notes TEXT,
  reason TEXT,
  captured_by_user_id TEXT NOT NULL,
  captured_at TEXT NOT NULL,
  UNIQUE(measurement_profile_id, version_no)
);

CREATE TABLE order_measurement_snapshots (
  id TEXT PRIMARY KEY,
  order_item_id TEXT NOT NULL,
  source_measurement_version_id TEXT,
  values_json TEXT NOT NULL,
  override_json TEXT,
  created_at TEXT NOT NULL
);
  • Updating the current measurement creates a new version, not an overwrite.
  • Order item stores snapshot values used at booking time.
  • One-off loose/tight changes are order overrides unless staff explicitly updates the base profile.
  • Photos belong in R2, referenced by metadata rows; never store image blobs in D1.

Phase 04 · Domain Model

Order, item, production, and payment model

Orders should support multiple garments and partial completion. A single order may include blouse, petticoat, alteration, and fall/pico work; each item can move through workflow independently.

CREATE TABLE orders (
  id TEXT PRIMARY KEY,
  order_code TEXT NOT NULL UNIQUE,
  contact_id TEXT NOT NULL REFERENCES customer_contacts(id),
  customer_profile_id TEXT NOT NULL REFERENCES customer_profiles(id),
  current_status TEXT NOT NULL,
  order_date TEXT NOT NULL,
  trial_date TEXT,
  promised_delivery_date TEXT,
  subtotal_paise INTEGER NOT NULL DEFAULT 0,
  discount_paise INTEGER NOT NULL DEFAULT 0,
  final_total_paise INTEGER NOT NULL DEFAULT 0,
  balance_due_paise INTEGER NOT NULL DEFAULT 0,
  created_by_user_id TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE order_items (
  id TEXT PRIMARY KEY,
  item_code TEXT NOT NULL UNIQUE,
  order_id TEXT NOT NULL REFERENCES orders(id),
  garment_type_code TEXT NOT NULL,
  quantity INTEGER NOT NULL DEFAULT 1,
  item_status TEXT NOT NULL,
  price_paise INTEGER NOT NULL DEFAULT 0,
  assigned_staff_user_id TEXT,
  promised_delivery_date TEXT,
  notes TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE payments (
  id TEXT PRIMARY KEY,
  payment_code TEXT NOT NULL UNIQUE,
  order_id TEXT NOT NULL REFERENCES orders(id),
  amount_paise INTEGER NOT NULL,
  mode TEXT NOT NULL,
  kind TEXT NOT NULL,       -- advance | balance | refund | correction
  reference TEXT,
  recorded_by_user_id TEXT NOT NULL,
  recorded_at TEXT NOT NULL
);

Phase 04 · Domain Model

Repository and service layer implementation

The repository layer hides SQL, while the service layer enforces business rules. This keeps route handlers thin and testable.

// Route → Service → Repository → D1
app.post('/v1/orders', zValidator('json', createOrderSchema), async (c) => {
  const result = await createOrderService({
    db: c.env.TENANT_DB,
    media: c.env.MEDIA_BUCKET,
    whatsappQueue: c.env.WHATSAPP_SEND_QUEUE,
    actor: c.get('user'),
    input: c.req.valid('json'),
    requestId: c.get('requestId')
  });
  return c.json({ ok: true, data: result, requestId: c.get('requestId') }, 201);
});

export async function createOrderService(ctx: CreateOrderContext) {
  return withTenantTransaction(ctx.db, async (tx) => {
    const customer = await customerRepo.findProfile(tx, ctx.input.customerProfileId);
    assertCanCreateOrder(ctx.actor, customer);
    const order = await orderRepo.insert(tx, buildOrderRecord(ctx.input, ctx.actor));
    await auditRepo.insert(tx, audit('ORDER_CREATED', ctx.actor, order.id));
    await outboxRepo.insert(tx, orderBookedEvent(order, ctx.requestId));
    return order;
  });
}

Phase 04 · Domain Model

Edge cases, tests, and checklist

Edge cases

  • Customer changes phone number: keep contact phone history; search old numbers as aliases.
  • Duplicate family profile: show possible matches and add merge workflow later.
  • Order created without measurement: allow only for alteration/simple service with explicit reason.
  • Refund after partial delivery: payment ledger must support refund and balance recalculation.
  • Measurement unit mismatch: show unit in every field and store unit in version.
  • Garment-specific schema evolves: version garment template and validate old measurements against old template.

Tests

  • Repository tests for contacts, profiles, measurement versions, orders, payments.
  • Service tests for create order, update measurement, receive payment, cancel order.
  • Migration tests using a fresh tenant DB and a seeded older schema.
  • Search projection tests after customer/order/payment updates.
  • Audit tests for payment, measurement, status, export, delete actions.

Exit checklist

  • Tenant DB migrations complete.
  • All high-traffic queries have indexes.
  • Domain services enforce invariants.
  • Outbox events are written for order booked, receipt generated, payment received, status changed.
  • Seed data includes garment types, statuses, roles, receipt numbering, measurement templates.
