WhatsApp Integration for TailorOS

Executive summary

Recommended WhatsApp approach

Build a separate WhatsApp Messaging Hub that TailorOS calls when it needs to send order confirmations, receipts, fitting reminders, ready-for-pickup alerts, delay messages, payment balance reminders, or customer status replies. The hub owns Meta credentials, template registry, webhook processing, delivery logs, and usage metering.

Architecture

Separate WhatsApp Messaging Hub

Do not place provider-specific WhatsApp code directly inside TailorOS screens. TailorOS should publish events and ask the Messaging Hub to send or receive messages. This lets the same WhatsApp product integrate with TailorOS, future CRM products, appointment apps, billing products, or other client projects.

TailorOS

  • Creates order events.
  • Stores customer and order context.
  • Shows notification status.
  • Receives inbound status-chat intents.

Messaging Hub

  • Stores tenant WhatsApp settings.
  • Chooses template and language.
  • Sends Cloud API requests.
  • Receives and validates webhooks.
  • Meters usage and costs.

Meta Cloud API

  • Accepts message requests.
  • Delivers WhatsApp messages.
  • Sends inbound messages and status webhooks.
  • Manages templates, phone number quality, and WABA state.
TailorOS event
  -> Messaging Hub send request
    -> tenant lookup
      -> template lookup
        -> Meta Cloud API /messages
          -> Meta webhook status
            -> Messaging Hub log update
              -> TailorOS notification status

Prerequisites

What you need before implementation

Meta account setup

  • Facebook account or managed Meta account.
  • Meta developer registration.
  • Meta Business Portfolio for your company or client shop.
  • Meta app with the WhatsApp use case.
  • WhatsApp Business Account, also called WABA.
  • Business phone number for WhatsApp Cloud API.
  • Test phone with WhatsApp enabled.

Technical setup

  • Public HTTPS webhook endpoint with valid TLS or SSL certificate. Self-signed certificates are not supported.
  • Backend service to send HTTP requests to Graph API.
  • Secure credential storage for access tokens and app secrets.
  • Database tables for tenants, WABAs, phone number IDs, templates, messages, webhooks, and usage.
  • Logging and retry-safe processing for webhooks.

Plain-language definitions

WABA is the WhatsApp Business Account. Phone Number ID is the ID Meta uses when your API sends messages from a specific WhatsApp number. Access token is the secret that allows your server to call Meta APIs. Webhook is the public URL where Meta sends incoming messages and delivery status updates.

Implementation steps

Step-by-step Meta Cloud API setup

1

Choose the onboarding model

For the first TailorOS pilot, manually onboard each shop in Meta Business tools. For a scalable multi-client SaaS, plan for Tech Provider onboarding and Embedded Signup later, so each shop can authorize its own WABA and phone number from your client portal.

  • Pilot mode: you help each shop create or connect a WABA manually.
  • SaaS mode: clients connect their WhatsApp account through a guided onboarding flow.
  • Important: do not put multiple unrelated tailor shops under one shared WhatsApp phone number.
2

Create a Meta app with WhatsApp

Open the Meta App Dashboard, create a new app, select the WhatsApp customer-communication use case, connect or create a business portfolio, and complete app creation. Meta's current quickstart flow redirects you to a WhatsApp quickstart or API setup screen.

  • App name and contact email should belong to your business.
  • Use a real business portfolio, not a personal placeholder, for production planning.
  • Keep a setup document with app ID, business ID, WABA ID, and phone number ID.
3

Connect a WhatsApp Business Account

In the API Setup section, connect the app to an existing WABA or create a new WABA. Save the WABA ID and the test phone number ID. The phone number ID is required for message sending.

Business ID WABA ID Phone Number ID Display phone number App ID
4

Generate a temporary token and send a test message

Use the dashboard's temporary access token to send the first test message from the test number to your own WhatsApp number. This proves that the app, WABA, phone number ID, recipient number, and basic messaging path work.

  • Temporary tokens expire quickly and are not suitable for production.
  • After receiving the test message, reply back from WhatsApp. This opens a 24-hour customer service window.
  • Retain the phone number ID and WABA ID after the test.
5

Create your webhook endpoint

Before production, Meta requires a public webhook endpoint that can answer verification GET requests and accept POST JSON payloads. The endpoint must use a valid TLS/SSL certificate.

GET verification behavior

GET <CALLBACK_URL>
  ?hub.mode=subscribe
  &hub.challenge=<HUB_CHALLENGE>
  &hub.verify_token=<HUB_VERIFY_TOKEN>
  • Store your own verify token on the server.
  • When Meta sends verification, compare hub.verify_token to your stored value.
  • If valid, return HTTP 200 with the raw hub.challenge value.
  • If invalid, return a 400-level response.

POST event behavior

POST <CALLBACK_URL>
Content-Type: application/json
X-Hub-Signature-256: sha256=<SHA256_PAYLOAD_HASH>

{ "object": "whatsapp_business_account", "entry": [...] }
  • Validate X-Hub-Signature-256 using HMAC-SHA256, the raw request body, and your Meta app secret.
  • Store every accepted payload or normalized event because there is no historical webhook API to fetch missed data later.
  • Return HTTP 200 quickly after validating and capturing the event.
  • Design idempotency because Meta can retry failed deliveries for up to 7 days and duplicates can happen.
6

Configure webhooks in the App Dashboard

In the WhatsApp configuration panel, add the Callback URL and Verify token. Once Meta verifies the endpoint, subscribe to the webhook fields your product needs.

Webhook field Why TailorOS needs it
messages Inbound customer messages and delivery status of messages sent by the business.
message_template_status_update Know when a template is approved, rejected, paused, or changed.
message_template_quality_update Monitor template quality and reduce block/report risk.
template_category_update Detect category changes that can affect pricing and compliance.
phone_number_quality_update Watch WhatsApp phone number health and throughput changes.
account_alerts Receive alerts about messaging limits, business profile, or official account status.
7

Create a system user and permanent token

The temporary token is only for testing. For development and production, create a system user in Business Settings, assign the app and WhatsApp account assets, and generate a system user access token.

  • Create a system user in Meta Business Settings.
  • Assign the app with Manage app or full control as required by the dashboard.
  • Assign the WhatsApp account with Manage WhatsApp Business accounts.
  • Generate a token with business_management, whatsapp_business_messaging, and whatsapp_business_management.
  • Store the token securely and never expose it in frontend code.
8

Send a service message inside the 24-hour window

When a customer messages the business, a 24-hour customer service window opens. During that window, you can send free-form service messages without pre-approved templates. Use this for customer-initiated status chat.

The current send-messages docs fetched for this guide show v25.0 in examples. Use the current Graph API version available in your Meta app and keep it configurable.

curl 'https://graph.facebook.com/v25.0/<PHONE_NUMBER_ID>/messages' \
  -H 'Content-Type: application/json' \
  -H 'Authorization: Bearer <SYSTEM_USER_ACCESS_TOKEN>' \
  -d '{
    "messaging_product": "whatsapp",
    "recipient_type": "individual",
    "to": "<CUSTOMER_WHATSAPP_NUMBER>",
    "type": "text",
    "text": {
      "preview_url": false,
      "body": "Your order ORD-2026-000881 is ready for pickup. Balance due: INR 450."
    }
  }'

The send API response only means Meta accepted the request. It does not prove delivery. Use messages webhooks to update sent, delivered, read, failed, and rejected status.

9

Create utility templates for order events

Outside the 24-hour service window, TailorOS must use approved templates. For tailor shop operations, most templates should be utility because they are non-promotional and tied to a customer order or request.

curl 'https://graph.facebook.com/v25.0/<WABA_ID>/message_templates' \
  -H 'Authorization: Bearer <SYSTEM_USER_ACCESS_TOKEN>' \
  -H 'Content-Type: application/json' \
  -d '{
    "name": "tailoros_order_confirmation",
    "category": "utility",
    "language": "en_IN",
    "parameter_format": "named",
    "components": [
      {
        "type": "body",
        "text": "Hi {{customer_name}}, your order {{order_id}} is confirmed. Delivery date: {{delivery_date}}. Advance paid: {{advance_amount}}. Balance: {{balance_amount}}.",
        "example": {
          "body_text_named_params": [
            { "param_name": "customer_name", "example": "Lakshmi" },
            { "param_name": "order_id", "example": "ORD-2026-000881" },
            { "param_name": "delivery_date", "example": "20 July 2026" },
            { "param_name": "advance_amount", "example": "INR 500" },
            { "param_name": "balance_amount", "example": "INR 450" }
          ]
        }
      }
    ]
  }'

    "to": "<CUSTOMER_WHATSAPP_NUMBER>",

11

Move from test number to production number

After testing, connect a real business phone number. The number should belong to the specific tailor shop, not the platform owner, unless you are running your own shop. Customers should recognize the number and business name.

  • Prepare business profile: display name, address, description, website if any.
  • Complete business verification if required for scale, limits, or account quality.
  • Confirm the number is not currently locked to an incompatible WhatsApp setup.
  • Submit templates early because approval can block launch.
  • Test inbound and outbound messages with real staff before go-live.
12

Launch with monitoring and fallback

For the first shops, monitor every message path. Keep a manual fallback button in TailorOS so staff can copy message content or call the customer when WhatsApp fails.

  • Dashboard: sent, delivered, read, failed, pending, rejected.
  • Alerts: token expiry, webhook failures, template rejection, low quality, high cost.
  • Usage: messages by client, category, template, recipient country, and period.
  • Fallback: staff can manually resend, switch to phone call, or mark customer notified.

Template strategy

TailorOS templates to create first

Keep every template specific, useful, and non-promotional. If you mix a discount, upsell, or festival offer into an order update, Meta may treat it as marketing. Marketing messages are useful later, but they are not the right default for order operations.

Template name Category Language Variables When to send
tailoros_order_confirmation Utility en_IN, ta customer_name, order_id, delivery_date, advance_amount, balance_amount Immediately after order creation.
tailoros_payment_receipt Utility en_IN, ta customer_name, receipt_id, order_id, paid_amount, balance_amount, receipt_link After advance, balance, or correction receipt.
tailoros_fitting_reminder Utility en_IN, ta customer_name, order_id, fitting_date, shop_name One day before or morning of fitting.
tailoros_ready_pickup Utility en_IN, ta customer_name, order_id, pickup_date, balance_amount, shop_name When order status becomes ready.
tailoros_delay_update Utility en_IN, ta customer_name, order_id, new_date, reason_short When promised date changes.
tailoros_balance_reminder Utility en_IN, ta customer_name, order_id, balance_amount, shop_name Before pickup or when staff requests reminder.
tailoros_delivery_confirmation Utility en_IN, ta customer_name, order_id, receipt_link After final delivery.

Language guidance

  • Meta's supported-language page lists Tamil code as ta.
  • For Indian English, use en_IN if you want India-specific English templates.
  • Meta does not translate template strings or variables. You must create Tamil text yourself.
  • Each language version counts separately against template limits.

Approval guidance

  • Use clear order-specific wording.
  • Do not include offers, coupons, festival sales, or upsell text in utility templates.
  • Include realistic example values for every variable.
  • Use named parameters where possible.
  • Subscribe to template status and quality webhooks.

Multi-client SaaS design

How to support many shops safely

Each client shop should have isolated WhatsApp settings and usage. A webhook payload includes identifiers such as WABA entry ID and phone number metadata. Use those identifiers to route events to the correct tenant.

Hub table Purpose Key fields
tenants Client product account, such as one tailor shop or one external app client. tenant_id, business_name, plan, status, billing_mode, default_language.
whatsapp_accounts Client's WABA and phone number mapping. tenant_id, provider, business_id, waba_id, phone_number_id, display_phone_number, quality_status.
provider_credentials Secure token and credential metadata. tenant_id, provider, token_reference, app_id, expiry, last_checked_at, active.
message_templates Approved template registry per tenant and language. tenant_id, provider_template_id, name, category, language, status, variables.
message_requests Outbound request created by TailorOS or another product. tenant_id, source_product, recipient, template, payload, idempotency_key, status.
message_events Webhook and delivery event log. tenant_id, provider_message_id, event_type, delivery_status, raw_payload_reference, occurred_at.
usage_records Monthly client usage and billing. tenant_id, month, category, country, count, estimated_cost, billed_status.
inbound_conversations Customer-initiated messages and status chat state. tenant_id, wa_id, customer_reference, open_window_until, current_intent, assigned_staff.

Routing logic

  • Use webhook entry.id and phone_number_id to locate the tenant.
  • Use wa_id or sender number to locate customer or family account in TailorOS.
  • When a phone number maps to many customer profiles, ask the customer to choose active order or customer.
  • Use idempotency keys to prevent duplicate sends from retries.

Credential isolation

  • Encrypt tokens and never show full tokens in UI.
  • Do not reuse one client's token for another client.
  • Restrict admin actions by tenant and role.
  • Record token changes and webhook configuration changes in audit logs.
  • Rotate tokens on a schedule or when staff access changes.

Future SaaS onboarding

For scale, investigate Meta's Tech Provider onboarding and Embedded Signup. Manual setup is fine for the first shops, but it becomes slow when you need dozens or hundreds of clients. Embedded Signup lets the customer authorize WhatsApp access from your portal, subject to Meta approval and app review requirements.

Pricing and cost controls

How WhatsApp billing affects TailorOS

WhatsApp Business Platform pricing is per delivered message. Meta's pricing page states that charging is based on market and message category. Categories are marketing, utility, authentication, and service. Service messages are free when replying inside the customer service window, and utility messages sent in response to users can also receive favorable treatment based on Meta's current pricing model.

Use low-risk categories first

  • Service: customer asks status, system replies during 24-hour window.
  • Utility: order confirmation, receipt, fitting reminder, ready alert, balance reminder.
  • Authentication: only needed if you add customer OTP login.
  • Marketing: festival offers and promotions later, only with opt-in and clear controls.

Cost controls

  • Send one confirmation, one receipt, one reminder, and one ready alert by default.
  • Do not send every internal status change to the customer.
  • Show estimated cost before bulk messages.
  • Meter usage by shop, category, and month.
  • Pause marketing if quality drops or costs spike.

Risks and safeguards

Common mistakes to avoid

High-risk mistakes

  • Putting access tokens in frontend JavaScript.
  • Ignoring webhook signature validation.
  • Assuming API accepted means message delivered.
  • Using marketing wording inside utility templates.
  • Sending customer data to the wrong family member under a shared phone number.
  • Using one WhatsApp number for many unrelated shops.
  • Not storing webhook events, then losing delivery history.

Safeguards

  • Backend-only token use.
  • HMAC validation for all POST webhooks.
  • Tenant isolation by WABA and phone number ID.
  • Audit logs for credential and template changes.
  • Template review before shop go-live.
  • Opt-in and opt-out tracking.
  • Manual fallback for failed message paths.

WhatsApp Connect product layer

Useful upgrades added after comparing the provider analysis

The separate WhatsApp product should be treated as WhatsApp Connect: a provider-neutral messaging layer that TailorOS can use first, and other future products can reuse later. Meta Cloud API direct remains the recommended starting provider, but the internal product should be designed so Twilio, 360dialog, or an Indian BSP can be added without rewriting TailorOS.

Recommended route

Start with Meta Cloud API direct for the first pilot because it gives the best cost control and teaches the core WhatsApp rules. Build the credential vault, template registry, webhook receiver, usage ledger, and provider adapter boundary from day one. Add Embedded Signup and optional BSP adapters only after the first few shops prove the workflow.

Provider option Best use case Pros Tradeoffs Recommendation
Meta Cloud API direct Cost-efficient SaaS foundation where you can handle setup, templates, webhooks, and support. Lowest platform markup, official API, full control, good long-term fit for multi-client SaaS. Requires Meta Business setup, app review paths, token management, webhook operations, and template discipline. Use first for pilot and core architecture.
Twilio WhatsApp Fast developer setup, teams already using Twilio, or customers needing SMS/voice fallback later. Excellent docs, mature tooling, easy sandbox, strong delivery logs, broad communication APIs. Usually adds Twilio markup and an extra provider layer; customer onboarding still depends on WhatsApp approval. Add as optional adapter after Meta direct.
360dialog Shops or agencies that want BSP-assisted WhatsApp Business onboarding with API access. Focused WhatsApp BSP model, can simplify onboarding and WABA operations for some clients. Additional provider relationship, pricing and support terms vary by plan and region. Evaluate for reseller-style onboarding after v1.
Indian BSPs such as Authkey, Interakt, or WATI Non-technical clients who value local support, campaign tools, inbox features, or managed onboarding. Local support can help Tamil Nadu SMBs; some include inbox, templates, and campaign UI. May be less flexible for a custom multi-product platform; markup, lock-in, and API behavior differ. Use only when a specific client needs managed setup or inbox features.

Core service responsibilities

  • Channel account manager: maps tenant, product installation, WABA, phone number, provider, quality rating, and messaging limit tier.
  • Credential vault: stores token references, webhook verify tokens, app secret references, expiry, rotation status, and audit history.
  • Template registry: tracks approved names, language, category, variables, provider status, rejection reason, and fallback copy.
  • Message orchestrator: checks opt-in, quiet hours, template availability, idempotency, rate limits, and send priority.
  • Webhook receiver: validates signature, normalizes provider payloads, deduplicates retries, and routes events to the correct tenant.
  • Conversation router: sends inbound chat to TailorOS, customer support, or a simple command bot.
  • Usage ledger: records delivered messages by tenant, country, category, template, and month for billing.
  • Admin console: helps support staff onboard clients, check failures, reconnect accounts, and review usage.

Multi-product tenant model

  • tenant: the client business account.
  • product_installation: TailorOS today, later SalonOS, RepairOS, ClinicOS, or another product.
  • whatsapp_channel_account: WABA, phone number ID, display number, provider, quality rating, messaging limit tier.
  • provider_credential: secret references only, never raw tokens in normal app tables or logs.
  • template_registry: provider template names mapped to internal template purposes such as order confirmation or payment receipt.
  • message_request: idempotent outbound request from any product.
  • message_event: provider status, inbound messages, failures, and delivery changes.
  • usage_record: monthly cost and billing evidence per tenant and product.
Internal API Purpose Required fields Important behavior
POST /messages/template Send a utility or marketing template from TailorOS or another product. tenant_id, product_id, recipient, template_purpose, language, variables, idempotency_key. Reject if opt-in is missing, template is not approved, or idempotency key already sent.
POST /messages/service Reply inside the customer service window after a customer messages first. tenant_id, conversation_id, recipient, message_body, source_product, staff_user_id if manual. Allow only when service window is open; otherwise suggest an approved template.
GET /conversations/{id} Show chat history in TailorOS without exposing provider-specific payloads. tenant_id, conversation_id, customer reference, order reference if known. Mask sensitive media by default and include provider delivery status on each outbound message.
POST /webhooks/{provider} Receive Meta, Twilio, or BSP webhook events. provider signature, provider account identifiers, message ID, event type, raw payload reference. Validate signature, store raw payload safely, normalize event, and process duplicates idempotently.
GET /usage Give owners and platform admins usage visibility. tenant_id, month, product_id, category, country, status. Show delivered count, failed count, estimated cost, billed status, and top templates.
Usage scenario Likely monthly messages Cost behavior Product action
Small tailor shop 100 orders, roughly 300 to 500 utility/service messages. Meta direct should remain inexpensive; fixed support time matters more than message cost. Bundle a simple monthly WhatsApp allowance and show usage only when near limit.
Growing shop or boutique 500 orders, roughly 1,500 to 2,500 messages. Template discipline matters because unnecessary reminders create avoidable cost and quality risk. Add usage dashboard, message caps, quiet hours, and template approval workflow.
Multi-shop SaaS platform 5,000+ order-related messages across many tenants. Per-tenant billing evidence, failures, quality rating, and provider routing become essential. Use usage ledger, provider adapters, support console, and automated onboarding checks.

Policy controls to add

  • Support STOP and a shop-approved local Tamil opt-out wording.
  • Do not send promotions from TailorOS order templates.
  • Separate utility templates from marketing templates in the registry.
  • Pause outbound messages for tenants with low quality rating or unresolved template rejections.
  • Never send full measurement photos or sensitive notes automatically.

WhatsApp Connect milestones

  1. Provider-neutral foundation: tenants, product installations, channel accounts, credentials, templates.
  2. Meta outbound templates: order confirmation, receipt, trial reminder, ready for pickup, payment due.
  3. Webhooks and delivery status: signature validation, dedupe, normalized events, message log.
  4. TailorOS event integration: map order and payment events to approved templates.
  5. Command bot and handoff: STATUS, RECEIPT, DUE, HELP, and staff assignment.
  6. Usage ledger and billing: monthly usage, estimated cost, allowance, overage, and export.
  7. Embedded Signup or BSP adapters: add only after pilot onboarding pain is proven.
