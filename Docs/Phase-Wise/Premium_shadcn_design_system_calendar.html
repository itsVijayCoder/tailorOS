Senior designer direction

The visual style should feel premium but not luxury-for-show. TailorOS is used in busy shops, so the design must be warm, fast, legible, and calm.

Warm golden surfaces Low-noise dashboards High-contrast actions Subtle motion Tamil-ready spacing Touch-first forms

Every status is understandableNo visual clutter
Every action gives feedbackNo guessing
Every tenant can brand itWithout rewriting UI

1. shadcn-compatible theme architecture

Use shadcn/ui’s semantic token model: components use background, foreground, primary, accent, border, ring, sidebar, and chart variables. A tenant theme becomes only a CSS-variable override, not a component rewrite.

Token groupTailorOS usageBest practice
background / foregroundApp shell, page background, default text.Keep background warm and calm; keep foreground dark enough for invoices and measurement screens.
card / popoverDashboard panels, order cards, command palette, calendar popovers, receipt drawer.Cards should not compete with the primary action; popovers should be slightly brighter and more elevated.
primaryMain CTA, selected nav, selected calendar date, premium brand highlight.Golden primary should usually use dark foreground for contrast, not white by default.
accent / mutedHover rows, subtle panels, empty states, helper text.Use for calm affordance; do not overuse gold for every row.
ring / border / inputFocus states, forms, search box, measurement fields.Focus ring must be visible on mobile and keyboard navigation.
sidebar tokensTenant navigation shell, active items, module grouping.Sidebar should feel branded but never reduce content readability.
chart tokensRevenue, ready/pending split, delivery performance, WhatsApp health.Charts should use semantic variables through chart config, not hardcoded hex.

3. Premium component system

Build TailorOS with shadcn components you own. Every component should use CSS variables and business state, not arbitrary Tailwind colour strings. The tenant should feel the product is custom-made for their shop.

Buttons

Primary for the next business action; outline for secondary; ghost for utility. Touch target 40–44px minimum.

Measurement fields

Compact but readable. Show unit, previous value, and validation hint for sensitive values.

4. Micro-interactions and page transitions

The animation system should make TailorOS feel fast and premium without slowing staff. Animate confirmation, navigation, and state changes. Do not animate everything.

Motion tokens

MotionDurationEasingUse
Button press90–120msease-outTap feedback, save, status update.
Hover lift120–180mspremiumCards, calendar events, nav rows.
Drawer/sheet180–260msspring-softReceipt drawer, order quick-view.
Page transition220–320mspremiumRoute content fade/translate; app shell stable.
Skeleton shimmer1.8–3.8slinear/easeOnly during actual loading.

Page transition design

Keep the sidebar/topbar persistent. Animate only the route content with opacity + small Y movement. Avoid sliding full pages from left/right because it feels heavy and can cause motion discomfort.

Enteropacity 0 → 1, y 8 → 0
Exitopacity 1 → 0, y 0 → -4
Reduced motionopacity only
packages/ui/src/animations/animated-page.tsxMotion implementation
"use client"

import { motion, useReducedMotion } from "motion/react"

export function AnimatedPage({ children }: { children: React.ReactNode }) {
  const reduceMotion = useReducedMotion()

  return (
    <motion.div
      initial={reduceMotion ? { opacity: 0 } : { opacity: 0, y: 8, scale: 0.995 }}
      animate={reduceMotion ? { opacity: 1 } : { opacity: 1, y: 0, scale: 1 }}
      exit={reduceMotion ? { opacity: 0 } : { opacity: 0, y: -4 }}
      transition={{ duration: 0.22, ease: [0.2, 0.8, 0.2, 1] }}
    >
      {children}
    </motion.div>
  )
}

6. Edge cases and verification checklist

Use this checklist before implementing the design system into the production repository.

Design-system checks

  • All colours come from shadcn semantic CSS variables or TailorOS semantic aliases.
  • Dark mode uses the same token names under .dark.
  • Tenant theme overrides only variables, not component source.
  • Buttons, cards, inputs, popovers, sidebars, charts and status chips are token-driven.
  • Focus ring is visible against light golden and dark themes.

Calendar checks

  • Month/week/day views are consistent and keyboard accessible.
  • Overdue, trial, pickup, delivery, WhatsApp and payment events are visually distinct.
  • Multiple order items on one order can appear separately but open one grouped drawer.
  • Tenant timezone is respected and selected date does not shift on hydration.
  • Mobile layout becomes agenda-first instead of squeezing a complex month grid.

Motion checks

  • Only transform and opacity are used for frequent motion.
  • Reduced-motion users get opacity-only transitions.
  • List polling does not reanimate all old rows.
  • Page shell remains stable while route content transitions.
  • No animation blocks save, status update, payment, or WhatsApp send actions.

Product edge cases

  • One mobile number can show many family/customer profiles without mixing measurements.
  • Payment corrections are visible and not hidden by pretty UI.
  • WhatsApp failed/blocked messages are visible on the order page, not only in admin logs.
  • Calendar filters do not hide overdue critical orders without a clear active filter state.
  • Print mode removes nav, actions and background gradients for clean receipts/reports.
