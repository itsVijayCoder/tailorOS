Token best practices

  • Use semantic classes in components: background, foreground, primary, muted, accent, destructive.
  • Keep tenant branding inside theme classes such as theme-gold, theme-sandal, theme-rose.
  • Do not hardcode colours in component JSX except one-off data visualisation values.
  • Use OKLCH tokens for predictable light/dark changes and better perceptual colour tuning.
  • Every semantic status must work in light and dark mode: ready, trial, overdue, payment, WhatsApp.
  • Use chart tokens for dashboards, not random palette utilities.

// packages/ui/src/components/order-status-chip.tsx
import { cva, type VariantProps } from "class-variance-authority";
import { cn } from "@/lib/utils";

const chipVariants = cva(
  "inline-flex items-center gap-1.5 rounded-full border px-2.5 py-1 text-xs font-medium transition-[transform,box-shadow,background-color] duration-150 ease-out hover:-translate-y-0.5",
  {
    variants: {
      status: {
        booked: "border-amber-200 bg-amber-50 text-amber-800",
        cutting: "border-blue-200 bg-blue-50 text-blue-800",
        stitching: "border-indigo-200 bg-indigo-50 text-indigo-800",
        trial: "border-fuchsia-200 bg-fuchsia-50 text-fuchsia-800",
        ready: "border-emerald-200 bg-emerald-50 text-emerald-800",
        overdue: "border-red-200 bg-red-50 text-red-800"
      }
    },
    defaultVariants: { status: "booked" }
  }
);

export function OrderStatusChip({ className, status, ...props }: React.ComponentProps<"span"> & VariantProps<typeof chipVariants>) {
  return <span className={cn(chipVariants({ status }), className)} {...props} />;
}

Motion and micro-interactions

Premium feel without slowing operations.

Page transition

180–260ms fade + 8px y shift. App shell stays stable.

Action feedback

Button compresses, toast enters, optimistic status chip updates.

Reduced motion

Respect user preference and keep information accessible.

Motion tokens

  • 90–140ms for button hover/press.
  • 160–220ms for drawers, popovers, command palette.
  • 180–260ms for route transitions.
  • Use transform and opacity for frequent animations.
  • Never delay order/payment save to complete an animation.

Micro-interaction map

  • Search palette: scale/fade entry, row highlight, keyboard hint.
  • Order status: chip colour + subtle x movement on timeline update.
  • Measurement save: checkmark, version number highlight, toast.
  • WhatsApp state: queued → sent → delivered → read with calm badge transitions.
  • Calendar day: hover elevation, selected day focus ring, side panel transition.

Design rules for developers

  • All components must use shadcn semantic tokens, not raw hex colours.
  • Keep server data in TanStack Query; UI state only in Zustand.
  • Every table/list item must have loading, empty, error, and permission states.
  • Calendar events must link to order, customer, payment, and WhatsApp log where relevant.
  • Every motion pattern must have reduced-motion behaviour.
  • Mobile layout must be designed first for counter use, not just squeezed desktop.
QA and design acceptance

Checklist to verify the product feels premium and works fast.

Visual QA

  • Gold, sandal, rose, emerald themes maintain contrast.
  • Dark mode is not just inverted; cards, borders, and chips remain readable.
  • All dashboards fit 1366px laptop and common Android mobile widths.
  • Receipts look printable and shareable.
  • Status colours are paired with text/icons.

UX QA

  • Owner can understand today’s workload in under 30 seconds.
  • Counter staff can search by mobile/order/customer ID immediately.
  • Calendar shows overdue risk before decorative metrics.
  • Every calendar event opens the related order or task.
  • WhatsApp failure states are visible where staff work.

Accessibility QA

  • Keyboard can reach nav, search, filters, calendar days, and drawer controls.
  • Focus ring is visible on light and dark themes.
  • Reduced-motion mode removes non-essential movement.
  • Text sizes remain readable on mobile.
  • Colour is never the only status cue.
