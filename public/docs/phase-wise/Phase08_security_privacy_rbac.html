Phase 08 · Security

Security, privacy, and tenant isolation objective

Phase 08 hardens TailorOS before real client data is trusted to it. The focus is tenant isolation, staff permissions, credential protection, signed media access, audit trails, public endpoint hardening, and safe support operations.

Non-negotiable rule. A bug must never allow one tailor shop to read another shop’s customers, measurements, receipts, WhatsApp messages, or credentials. Tenant isolation is a product feature, not only an implementation detail.

export async function requireTenantMembership(c: Context, next: Next) {
  const tenant = await resolveTenantFromHostOrPath(c.req.url, c.env.CONTROL_DB);
  const user = await requireSession(c.req, c.env);
  const membership = await membershipRepo.find(c.env.CONTROL_DB, tenant.id, user.id);

  if (!membership || tenant.status !== 'active') {
    return c.json(error('FORBIDDEN', 'You cannot access this shop'), 403);
  }

  c.set('tenant', tenant);
  c.set('user', { ...user, role: membership.role });
  await next();
}

Phase 08 · Security

WhatsApp credentials and secret handling

Tenant WhatsApp credentials are dynamic and should be encrypted in the backend, not committed as environment variables. Platform-level keys live as Workers secrets. Tenant access tokens are stored encrypted with rotation metadata and never returned to the browser.

  • Store only encrypted access token and metadata in D1.
  • Keep master encryption key in Worker secret / secret store.
  • Hash verify tokens if raw value is not needed.
  • Mask phone number IDs and business IDs in normal UI if not required.
  • Log credential changes, not credential values.
  • Support health checks report “valid/invalid/expired/permission issue” without exposing secrets.

Phase 08 · Security

Public endpoint hardening and abuse prevention

  • Signup: Turnstile, IP/email/mobile rate limits, slug reservation idempotency.
  • Login: rate limit by account and IP, secure cookies, failed login audit.
  • Search API: authenticated only, min length, user/tenant limits, no wildcard dump.
  • Receipt links: signed token, expiry, optional mobile-last-4 or order-code confirmation.
  • WhatsApp webhook: verify challenge, signature validation, dedupe, quick response, async queue processing.
  • Exports: owner-only, generated asynchronously, expiring signed links.

Phase 08 · Security

Security tests and launch checklist

Tests

  • Cross-tenant access attempt is rejected even with valid session from another tenant.
  • Tailor role cannot open payment reports or exports.
  • Raw WhatsApp token never appears in API responses, logs, or browser bundle.
  • Invalid webhook signature is rejected before queueing.
  • Receipt link expires and cannot be guessed.
  • Support access requires reason and creates audit row.

Exit checklist

  • RBAC matrix implemented in code and tests.
  • Tenant resolution does not trust request body tenant ID.
  • Credential encryption completed.
  • Audit log covers measurement, payment, status, export, delete, credential changes.
  • Public endpoint rate limiting in place.
  • Data export/delete runbooks documented.
