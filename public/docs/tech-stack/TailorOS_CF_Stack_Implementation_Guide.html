1. Executive decision

The best implementation for your requirement is a Cloudflare-first multi-service monorepo: Next.js for the web app, Hono Workers for APIs, a control-plane D1 database, one D1 database per tenant, R2 for media/receipts, Queues for background jobs, Durable Objects for realtime chat later, and a separate WhatsApp Connector service with provider adapters.

Recommended architecture name: Cloudflare Tenant-Isolated SaaS Pattern. Use one global tailoros_control D1 database to manage tenants, plans, domains, staff identity, provisioning state, integration metadata, and worker routing. For each new tailor shop, provision a separate D1 database such as tailoros_tenant_madurai_sri_raja_01h.... The tenant database stores that shop’s customers, family members, measurements, orders, payments, receipts, search index, WhatsApp logs, and audit logs.

Why this is the right direction

RequirementRecommended implementationReason
Fast app Next.js on Cloudflare Workers + Static Assets Static shell and edge-executed API routes reduce latency for users in India and globally.
Low cost Workers, D1, R2, Queues Cloudflare pricing is usage-oriented and scales to zero; no always-on server for the MVP.
Separate data per client Per-tenant D1 database Better isolation, easier export/delete/backup, lower blast radius, simpler per-shop scaling.
Multiple WhatsApp clients WhatsApp Connector Worker + tenant credentials table One reusable connector can support many shops with different WhatsApp Business credentials.
Quick search D1 indexes + FTS5 + tenant-local search projection Exact mobile/order search stays indexed; fuzzy/name/item search uses a dedicated search table.
Smooth UX shadcn/ui + Motion + CSS transform/opacity animations Modern micro-interactions without blocking user actions or harming performance.
Important correction for the “one D1 DB per tenant” idea. You should not simply add every tenant D1 database as a static binding to one Worker forever. D1 has database and binding limits, and runtime D1 access is designed around Worker bindings. For true self-serve tenant onboarding, use a control plane + provisioning queue + dispatch/tenant Worker pattern. For the first 5–10 pilot shops, you can manually bind databases, but for serious SaaS you should automate it.

Control-plane vs tenant-plane

Control plane

Global platform data lives here: tenants, tenant domains, owner accounts, plan, provisioning status, D1 database UUID/name, tenant worker name, WhatsApp integration registry, global audit events, billing metadata, and support/admin tools.

Tenant plane

Shop-specific operational data lives here: contacts, family customer profiles, measurements, orders, order items, receipts, WhatsApp message history, tenant-local search index, and shop audit logs.

This split matters because a tailor shop’s operational data should not be joined with another shop’s data. It also makes tenant export, tenant deletion, backup restore, and high-volume tenant migration much cleaner.

3. Cloudflare service map

Use Cloudflare services by their strongest role. Do not use one service for everything. The table below shows the recommended mapping for TailorOS.

Workers API, web app runtime, webhooks, background consumers Keep Workers small and composable; use bindings and service bindings. Long CPU-heavy processing, large video/image pipelines.
Workers Static Assets Next.js/static assets deployment Prefer for new Worker-based web deployments where possible. Private file storage.
D1 Control DB and per-tenant relational DB Use indexes, prepared statements, migrations, row-count metrics, read replication sessions. Large file blobs, analytics warehouse, unlimited-size relational DB.
R2 Receipts, garment reference photos, design images, generated PDFs Use signed upload/read URLs, CORS restrictions, object key namespacing. Searchable relational data or hot counters.

Key Cloudflare best practice. Use platform bindings for D1, R2, KV, Queues, and internal Worker calls. Bindings are direct in-process references exposed on env; they avoid the overhead and complexity of calling Cloudflare REST APIs for normal runtime operations.

4. D1 per-tenant database strategy

Cloudflare documents D1 as suitable for horizontal scale-out across many smaller databases, including per-user, per-tenant, and per-entity patterns. That matches your requirement: when a tailor shop registers, create a separate D1 database for that shop.

Recommended D1 topology

CONTROL_DB: tailors_control_prod ├─ tenants ├─ tenant_domains ├─ tenant_database_registry ├─ tenant_worker_registry ├─ users ├─ memberships ├─ integration_accounts ├─ provisioning_jobs └─ platform_audit_logs TENANT_DB: tailors_tenant_{slug}_{shortid} ├─ customer_contacts ├─ customer_profiles ├─ measurement_profiles ├─ measurement_versions ├─ orders ├─ order_items ├─ order_status_history ├─ payments ├─ receipts ├─ r2_files ├─ search_docs / search_fts ├─ wa_messages ├─ wa_webhook_events ├─ outbox_jobs └─ audit_logs

Why a control DB is mandatory

With one D1 database per tenant, the app needs a central registry that tells the platform:

  • Which tenant owns which D1 database.
  • Which tenant worker should handle that shop’s requests.
  • Which domain/subdomain maps to which tenant.
  • Which schema version a tenant database is running.
  • Whether provisioning, migration, suspension, export, or deletion is in progress.
  • Which WhatsApp phone number ID maps to which tenant integration.

D1 limits that influence the design

Limit areaWhy it mattersDesign response
Databases per account The paid account limit supports many databases, while the free account allows only a small number. Use free tier for development/pilot only. Per-tenant database SaaS needs a paid Cloudflare plan once you exceed pilot scale.
Database size D1 databases are intentionally smaller than a giant Postgres cluster. Use one DB per tenant; keep files in R2; archive old WhatsApp events; aggregate analytics separately.
Worker bindings A Worker script has a practical binding limit, so one Worker with thousands of static D1 bindings is not the clean long-term answer. Use Workers for Platforms dispatch namespaces or controlled per-tenant worker deployment.
Cross-tenant joins You cannot easily perform SQL joins across thousands of separate tenant DBs. Use event-driven projections into a platform analytics/search DB for admin-wide reporting.
FTS virtual table export D1 supports FTS5, but virtual tables need care during exports/migrations. Store canonical data in normal tables and rebuild FTS indexes during restore/migration.
Free-tier reality. A strict “new tenant = new D1 database” model is not ideal for a completely free-tier SaaS beyond a small pilot, because the free plan’s database count/storage limits are intentionally small. For 1–10 pilot shops, it works. For 25+ self-serve shops, plan for paid Cloudflare usage or start with shared-D1 development and migrate tenants before launch.

Primary location and read replication

For Tamil Nadu users, set D1’s broad primary location hint to apac when creating tenant databases. This is not a Chennai-specific guarantee, but it is the closest broad location hint exposed by the D1 create API. Enable D1 read replication with read_replication: { "mode": "auto" } for read-heavy tenants where available. When using D1 read replicas, use the D1 Sessions API so sequential user actions remain consistent.

Runtime access: bindings first, API second

For hot application traffic, use D1 bindings inside Workers. Use the Cloudflare D1 API only for administrative operations such as creating a tenant database, checking metadata, or applying migrations in a controlled provisioning flow.

OperationUseReason
Customer search, order create, payment update env.TENANT_DB.prepare(...) Low-latency binding path designed for Workers.
Create a new tenant database Cloudflare D1 REST API from control-plane Worker Administrative provisioning operation.
Apply tenant migration Wrangler/CI or admin D1 API Controlled and auditable schema changes.
Bulk analytics across all tenants Queue-driven projection or export pipeline Avoid scanning every tenant DB from a user request.

5. New tenant provisioning flow

Tenant onboarding must be asynchronous. Do not create a D1 database, apply migrations, deploy worker bindings, seed data, and connect WhatsApp inside a single user-facing HTTP request. Use a provisioning job.

Self-serve registration flow

  1. Signup submitted: shop owner enters shop name, city, owner name, mobile number, email, preferred subdomain, and plan.
  2. Abuse protection: validate Turnstile token server-side before creating anything.
  3. Control record: insert tenant row with status provisioning_pending.
  4. Slug reservation: reserve {slug}.tailoros.in or route path /t/{slug}.
  5. Provisioning job: send a message to tenant-provisioning-queue.
  6. Create D1: control-plane Worker calls Cloudflare D1 create API with APAC location hint.
  7. Apply migrations: run base tenant SQL migrations and record schema version.
  8. Seed defaults: add garment categories, status workflow, receipt numbering, measurement templates, default roles.
  9. Bind/deploy tenant worker: deploy or register a tenant Worker with that tenant’s D1 binding.
  10. Activate: update control DB status to active and notify owner.

Provisioning state machine

requested → validating_owner → tenant_reserved → db_creating → db_migrating → defaults_seeding → worker_deploying → active failure states: → failed_validation → failed_db_create → failed_migration → failed_worker_deploy → needs_manual_review

Provisioning code sketch

// apps/control-plane/src/jobs/provisionTenant.ts
export async function provisionTenant(job: ProvisionTenantJob, env: Env) {
  const tenant = await getTenantForProvisioning(env.CONTROL_DB, job.tenantId);

  if (!tenant || tenant.status !== "provisioning_pending") {
    return; // idempotent no-op
  }

  await markProvisioningStep(env.CONTROL_DB, tenant.id, "db_creating");

  const dbCreateResponse = await fetch(
    `https://api.cloudflare.com/client/v4/accounts/${env.CF_ACCOUNT_ID}/d1/database`,
    {
      method: "POST",
      headers: {
        Authorization: `Bearer ${env.CF_API_TOKEN}`,
        "Content-Type": "application/json"
      },
      body: JSON.stringify({
        name: tenant.dbName,
        primary_location_hint: "apac",
        read_replication: { mode: "auto" }
      })
    }
  );

  if (!dbCreateResponse.ok) {
    await failProvisioning(env.CONTROL_DB, tenant.id, "failed_db_create");
    throw new Error("D1 database creation failed");
  }

  const created = await dbCreateResponse.json();

  await saveTenantDatabaseRegistry(env.CONTROL_DB, {
    tenantId: tenant.id,
    dbName: tenant.dbName,
    dbUuid: created.result.uuid,
    readReplication: created.result.read_replication?.mode ?? "disabled"
  });

  await markProvisioningStep(env.CONTROL_DB, tenant.id, "db_migrating");

  // Apply migrations through CI/admin API, not user request path.
  await env.TENANT_MIGRATION_QUEUE.send({
    tenantId: tenant.id,
    dbUuid: created.result.uuid,
    targetVersion: CURRENT_TENANT_SCHEMA_VERSION
  });
}

Workers for Platforms pattern

For a scalable dynamic tenant system, use a dispatch namespace. A dynamic dispatch Worker resolves the tenant from hostname/path/header, performs platform-level checks, then invokes the tenant Worker with env.DISPATCHER.get(workerName). 

// apps/api-gateway/src/dispatch.ts
export default {
  async fetch(request: Request, env: Env, ctx: ExecutionContext) {
    const url = new URL(request.url);
    const tenantSlug = resolveTenantSlug(url.hostname, url.pathname);

    const tenant = await findTenantBySlug(env.CONTROL_DB, tenantSlug);
    if (!tenant || tenant.status !== "active") {
      return new Response("Tenant not found", { status: 404 });
    }

    await assertUserCanAccessTenant(request, tenant.id, env);

    const tenantWorker = env.DISPATCH_NAMESPACE.get(tenant.worker_name);
    return tenantWorker.fetch(request);
  }
};

6. TailorOS product structure

TailorOS should be built around real tailoring workflows, not just generic CRUD. The core domain is: contacts, family customer profiles, measurement versions, orders, payments, delivery status, receipts, WhatsApp communication, and audit history.

Important customer identity rule

Mobile number is not the customer ID. A single mobile number can belong to a family/contact, and multiple customer profiles can exist under it. Example: one mobile number can have profiles for father, mother, son, daughter, and each profile gets a unique customer code and measurement history.
Mobile: +91 98765 43210 │ ├─ Contact ID: CON-01HV... │ ├─ Customer profile: CUS-MDU-000231 Rajendran ├─ Customer profile: CUS-MDU-000232 Meena ├─ Customer profile: CUS-MDU-000233 Karthik └─ Customer profile: CUS-MDU-000234 Ananya Each customer profile: ├─ Measurement profile(s) ├─ Measurement version history ├─ Orders ├─ Receipts └─ WhatsApp communication through contact/mobile

Core product modules

ModuleFeaturesNotes
Dashboard Today’s deliveries, pending trials, unpaid balances, new messages, delayed orders. Should be the first screen for shop staff.
Contacts & family Mobile search, WhatsApp opt-in, address, family profiles, relationship labels. Mobile uniqueness should be per tenant, not global.
Customer profiles Name, gender/age group, customer code, notes, preferences, default measurement templates. Editable but audit important fields.
Measurements Garment-specific templates, current measurement, version history, reference photos. Never overwrite without versioning.
Orders Garment items, quantities, fabric info, trial date, delivery date, status workflow. Each order and item should have unique IDs.
Payments Estimate, advance, balance, partial payments, refunds, payment mode. Keep payment ledger separate from order total.
Receipts Printable receipt, WhatsApp receipt link, PDF/image receipt, receipt number. Receipt files stored in R2; metadata in D1.
WhatsApp inbox Inbound messages, order status query, staff replies, template sends. Use polling first; Durable Objects later for realtime.
Reports Daily revenue, delivery performance, pending balances, category sales. Per tenant; aggregate later via projections.
Settings Shop info, receipt format, garment templates, roles, WhatsApp integration. Keep template mapping editable by tenant admin.

Order status workflow

Use an explicit state machine instead of random text statuses. This enables consistent search, reports, WhatsApp messaging, and staff workflows.

draft → confirmed → fabric_received → cutting → stitching → trial_ready → trial_done → alteration_needed → ready_for_delivery → delivered side states: → on_hold → cancelled → refunded

7. WhatsApp Connector structure

The WhatsApp integration should be a separate reusable product layer. TailorOS is one client of this connector, but the same connector should support other future SaaS products and multiple tailor shops with their own WhatsApp Business credentials.

Connector boundaries

WhatsApp Connector deployables

whatsapp-connector

Public webhook endpoints for Meta and private APIs for sending messages. Validates signatures, maps phone number IDs to tenants, normalizes event payloads, and sends jobs to queues.

whatsapp-consumer

Queue consumer that sends template/text/media messages, handles retries, writes provider responses, and updates tenant DB message state.

packages/whatsapp

Provider adapter library. Starts with Meta Cloud API direct, later supports Twilio, 360dialog, Authkey, WATI/Interakt-style providers.

packages/message-policy

Rules for service window, template requirement, customer opt-in, rate limits, retry policy, and Tamil/English template variables.

8. Fast global search design

“Global search” should mean two things: a fast tenant-wide command search for staff, and an optional platform-wide admin/support search. The per-tenant D1 architecture makes tenant-wide search fast and isolated, but platform-wide search requires a projection/analytics pattern.

Tenant-wide search experience

The staff should press Ctrl + K or tap the search bar and find: mobile number, customer name, customer ID, order ID, receipt number, garment type, delivery date, and WhatsApp message.

CUS-MDU-000231 Exact customer profile Unique customer code index.

today delivery Delivery list for current date Status/date index and shortcut parser.

Search query strategy

  1. Normalize first: remove spaces, country-code normalize mobile numbers, uppercase order/customer codes.
  2. Fast exact checks: if input looks like mobile/order/customer code, run indexed exact or prefix query first.
  3. FTS fallback: if input is name/item/free text, use FTS5 against search_docs.
  4. Limit results: return 5–10 per category, not 100 rows.
  5. No heavy offset pagination: use cursor/keyset pagination for lists.
  6. Debounce frontend: 120–180ms debounce, minimum 2 characters, cancel stale queries.

API example

// packages/search/src/searchTenant.ts
export async function searchTenant(db: D1Database, rawQuery: string) {
  const q = normalizeSearchQuery(rawQuery);

  if (q.kind === "mobile") {
    return db.prepare(`
      SELECT 'contact' AS entity_type, id, mobile_e164 AS title, city AS subtitle
      FROM customer_contacts
      WHERE mobile_e164 LIKE ?1
      LIMIT 10
    `).bind(`${q.e164Prefix}%`).all();
  }

  if (q.kind === "order_code") {
    return db.prepare(`
      SELECT 'order' AS entity_type, id, order_code AS title, current_status AS subtitle
      FROM orders
      WHERE order_code = ?1
      LIMIT 1
    `).bind(q.value).all();
  }

  return db.prepare(`
    SELECT entity_type, entity_id, title, subtitle, bm25(search_docs) AS rank
    FROM search_docs
    WHERE search_docs MATCH ?1
    ORDER BY rank
    LIMIT 20
  `).bind(q.ftsQuery).all();
}

Platform-wide support search

Because each tenant has a separate D1 database, admin/support search across all tenants should not query every tenant database live. Instead, whenever a tenant creates or updates an important record, publish a queue event to update a small global projection in the control database or a dedicated analytics/search database.

Tenant DB write → Queue event: search_projection_update → Projection Worker → CONTROL_SEARCH_DB / analytics DB → Platform admin search endpoint
Do not use KV as the source of truth for search. KV is useful for cached search suggestions and feature flags, but it is eventually consistent. Authoritative customer/order/payment/search state should live in D1 or an explicit search service.

TanStack Query pattern

export function useCustomerSearch(query: string) {
  return useQuery({
    queryKey: ["search", "customers", query],
    queryFn: () => api.search.customers({ q: query }),
    enabled: query.trim().length >= 2,
    staleTime: 20_000,
    gcTime: 5 * 60_000
  });
}

export function useUpdateOrderStatus() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: api.orders.updateStatus,
    onMutate: async (input) => {
      await queryClient.cancelQueries({ queryKey: ["order", input.orderId] });
      const previous = queryClient.getQueryData(["order", input.orderId]);

      queryClient.setQueryData(["order", input.orderId], (old: Order | undefined) =>
        old ? { ...old, currentStatus: input.status } : old
      );

      return { previous };
    },
    onError: (_error, input, context) => {
      queryClient.setQueryData(["order", input.orderId], context?.previous);
    },
    onSettled: (_data, _error, input) => {
      queryClient.invalidateQueries({ queryKey: ["order", input.orderId] });
      queryClient.invalidateQueries({ queryKey: ["orders"] });
    }
  });
}

Frontend performance rules

  • Use route-level code splitting and lazy-load heavy drawers, report charts, and inbox panels.
  • Use skeleton states for list pages and order/customer detail panels.
  • Prefetch likely next pages: search results, order details, customer detail after hover/touch intent.
  • Use optimistic UI for status updates, but always reconcile after server response.
  • Virtualize long order/customer lists if they exceed 100 visible rows.
  • Keep server data in TanStack Query, not Zustand.
  • Use form-level Zod validation and server-side Zod validation; never trust client validation only.
  • Use a PWA manifest and service worker later for offline-friendly shop use.

11. Micro-interactions & smooth page transitions

TailorOS should feel premium and responsive, but the animations must be practical. The rule is: animate feedback, not decoration. Use motion to confirm actions, clarify state changes, and reduce perceived waiting time.

Animation principles

PrincipleImplementation
Respect accessibility Use prefers-reduced-motion and Motion’s reduced-motion utilities.
Use cheap properties Prefer transform and opacity; avoid animating layout-heavy properties.
Keep durations short Buttons: 90–140ms. Drawer/modal: 160–220ms. Page transition: 180–260ms.
Do not block work Animations should not delay order creation, receipt generation, or message send.
Make status visible Use animated chips/timelines for order progress and WhatsApp delivery states.

Receipt slides in from right; share/print actions appear after content loads.

Queued → sent → delivered → read indicators update without page reload.

Dashboard route content fades and moves 6–10px; app shell remains stable.

    </motion.div>
  );
}

Internal service binding pattern

Public browser requests should go through the API Gateway. Internal Worker-to-Worker calls should use service bindings, not public URLs. This keeps internal services private and avoids extra network overhead. 

// api-gateway calls whatsapp-connector without public Internet round-trip
await env.WHATSAPP_CONNECTOR.fetch(

Queue practices

  • Separate queues by job type: wa-send, wa-webhook, receipt-generate, tenant-provision, tenant-migrate.
  • Use DLQs for failed WhatsApp sends, failed webhooks, and failed provisioning jobs.
  • Every queue job must have an idempotency key.
  • Do not assume message ordering.
  • Keep message payloads small; store large payloads in D1/R2 and pass IDs in the queue message.
  • Explicitly acknowledge successful messages in batches to avoid retrying the whole batch when one fails.

Idempotency table

CREATE TABLE idempotency_keys (
  key TEXT PRIMARY KEY,
  route TEXT NOT NULL,
  request_hash TEXT NOT NULL,
  response_json TEXT,
  status TEXT NOT NULL,       -- processing | completed | failed
  expires_at TEXT NOT NULL,
  created_at TEXT NOT NULL
);

D1 query practices

  • Always use prepared statements with bound parameters.
  • Add explicit indexes for every high-traffic predicate.
  • Measure rows_read and rows_written from D1 query metadata.
  • Use D1 Sessions API when read replication is enabled and a user needs sequential consistency.
  • Avoid scanning WhatsApp logs or audit logs in normal dashboard queries.
  • Archive or compact high-volume webhook logs after retention period.

13. Security and tenant isolation

Multi-tenant SaaS security must be designed from day one. The goal is to prevent cross-tenant data access, credential leakage, unauthorized WhatsApp sends, and abuse of public endpoints.

Tenant isolation rules

RuleImplementation
Tenant chosen by platform, not user input Resolve tenant from verified domain/subdomain/path and authenticated membership.
Per-tenant DB binding Tenant Worker receives only its own TENANT_DB binding.
No browser access to credentials WhatsApp tokens and app secrets never leave backend Workers.
Role-based authorization Owner, manager, cutter, tailor, cashier, viewer roles with route/action checks.
Audit sensitive actions Measurement edit, order cancellation, payment edit, receipt delete, WhatsApp credential change.
Signed private file access R2 receipts/photos accessed through signed URLs or authenticated streaming Worker.

Recommended roles

RolePermissions
Owner All access, billing, integrations, staff management, export/delete.
Manager Customers, orders, measurements, payments, reports, WhatsApp inbox.
Tailor/Cutter View assigned orders, update work status, view measurements, add production notes.
Cashier Create receipts, collect payments, view balances, print bills.
Viewer Read-only dashboard and customer/order lookup.

Secrets and credential management

  • Use Workers secrets for platform-level secrets: Cloudflare API token, JWT/session keys, encryption master key, Meta app secret.
  • Do not put tenant WhatsApp access tokens in vars or committed files.
  • Encrypt tenant integration credentials before writing to D1.
  • Use separate staging and production credentials.
  • Rotate WhatsApp tokens and record token_last_rotated_at.
  • Hash webhook verify tokens; never store raw verify token if not needed.

Public endpoint hardening

EndpointProtection
Signup Turnstile server validation, rate limit by IP/mobile/email, duplicate tenant slug check.
Login Rate limit by IP and account, secure cookie, audit failed attempts.
Search Authenticated only, min query length, debounce, rate limit by user/tenant.
Receipt public link Signed token, expiry, optional mobile-last-4 confirmation.
WhatsApp webhook Provider verification/signature validation, dedupe, quick response, async queue processing.

  • Use preview deployments for frontend changes; use staging Worker env for backend changes.
  • Use separate Cloudflare API tokens with least privilege for CI.

16. Edge cases and product-quality checklist

The following edge cases are where TailorOS can become much better than a simple tailor billing app.

Edge caseRiskRecommended handling
Same mobile number, many people Wrong measurements/order assigned to wrong family member. Always select customer profile under contact. Show profile chips before creating an order.
Customer changes mobile number Old order history becomes hard to find. Keep contact history table and allow multiple numbers per contact later.
Duplicate customer creation Fragmented family/order history. Before save, show possible matches by mobile/name. Allow merge workflow.
Measurement correction Old orders may need original measurement. Version measurements; order item should reference measurement version used.
Partial order delivery One order has multiple garments, some delivered, some pending. Track order item status separately from order status.
Advance payment refund Balance calculations become wrong. Use payment ledger rows; never only store one mutable paid amount.
WhatsApp duplicate webhook Message/status processed twice. Dedupe using provider event/message IDs.
WhatsApp status arrives out of order Read event arrives before delivered event. Status rank model: queued < sent < delivered < read; never downgrade except failed terminal cases.
Template not approved Order notification fails silently. Template health screen in settings; queue failure visible in admin inbox.
Tenant DB migration fails Some tenants on old schema, some on new. Record per-tenant schema version and feature-gate by version.
D1 DB size growth Tenant hits DB size limit. Store files in R2, archive raw webhook logs, compact audit logs, move large tenant to dedicated plan/Postgres if needed.
Cross-tenant support search Per-tenant DBs cannot be queried in one SQL join. Use global projection DB updated by queues.
Offline/poor internet shop Staff cannot create order during network drop. Later PWA offline draft queue with conflict resolution. MVP: autosave drafts locally.
Heavy PDF generation Worker runtime may not suit complex PDF rendering. Start with HTML receipt + print. Offload advanced PDF generation to a small service later if needed.
Data deletion request Customer privacy issue. Per-contact anonymization/deletion workflow; retain financial records according to business rules.

Production readiness checklist

  • High: Tenant isolation tested with automated tests.
  • High: WhatsApp credentials encrypted and never exposed to browser.
  • High: Queue DLQs configured and monitored.
  • High: D1 migrations reversible or repairable.
  • High: Customer/mobile/order search tested with Tamil/English names and partial mobile input.
  • Medium: R2 object lifecycle/retention policy documented.
  • Medium: Tenant export and delete scripts tested.
  • Medium: Animation respects reduced-motion preference.
  • Later: Durable Object realtime inbox and order board.
  • Later: PWA offline drafts and background sync.
